  - `framework-oauth-client` : **소셜 로그인(OAuth2) + OIDC RP 강화** — 외부 IdP 인가코드 흐름 → `OAuthUserResolver`(앱) 매핑 → security 로 자체 JWT. google/kakao/naver 프리셋, state memory\|redis, OIDC(per-provider) id_token 검증. **새 외부 의존성 0**. 토글 `framework.oauth-client.enabled=false`

Spring Security SAML2 SP + IdP 메타데이터 등록(`RelyingPartyRegistrations.fromMetadataLocation`) → ACS 성공 시 `SamlUserResolver`(앱, `OAuthUserResolver` 대칭) 매핑 → `SamlTokenIssuer`(security `JwtProvider`/`TokenStore` 재사용)로 **자체 JWT 즉시 발급**(서버 세션 없이, 수기 JSON). 전용 `SecurityFilterChain`(`/saml2/**`,`/login/saml2/**` + 높은 우선순위, `@AutoConfiguration(after=SecurityAutoConfiguration)`)으로 framework-security 무수정. ⚠️ **이 프레임워크 최초의 "새 외부 의존성 0" 예외**: `spring-security-saml2-service-provider`+전이 OpenSAML(버전=SS 관리, 핀 X). **OpenSAML 4+ 는 Central 밖 → 루트에 Shibboleth 저장소(그룹 한정) 필수.** 멀티 파드=현재 게이트웨이 스티키 세션(redis AuthnRequest 저장소는 후속, redis 설정 시 fail-fast). 토글 `framework.saml-sp.enabled=false`

**서비스** — `services/`
- `gateway` (:8000, WebFlux+Resilience4j) · `user-service` (:8080) · `admin-service` (:8081)

## 2. 핵심 설계 원칙
- **계약은 공통, 구현은 프로젝트**: 예) 인증은 `Authenticator` 인터페이스만 공통, 프로젝트가 구현(user-service 의 `DbAuthenticationProvider`). 구현 빈이 있으면 `AuthAutoConfiguration` 이 공통 로그인(`LoginService`+`AuthController`)을 자동 활성화.
- **빈 등록은 `@AutoConfiguration` + `.imports`** (컴포넌트 스캔 밖). 신규 프레임워크 빈도 동일 패턴. 프로젝트는 같은 타입 빈을 등록해 `@ConditionalOnMissingBean` 으로 덮어쓸 수 있다.
- **선택형 모듈은 의존성 추가로 활성화**: 모듈을 의존성에 넣고 `type` 류 프로퍼티로 켠다(예: TokenStore/LoginAttempt 의 `type=redis`). property 상호배제라 오토컨피그 순서에 의존하지 않음.
- **동적 인가(RBAC)는 DB 기반**: `resources`/`role_resources` 매핑을 `DynamicAuthorizationManager` 가 평가. **매핑이 없으면 인증된 사용자에게 허용**(deny-by-default 아님)에 유의.

## 3. 보안 골격 (현재 상태)
- **JWT + TokenStore**: access(jti 부여)/refresh(1회용 회전), 로그아웃 시 jti 블랙리스트. TokenStore `type`: memory(기본)/redis/jdbc. 권한 authority 는 `ROLE_` 접두사(JwtProvider 부여, DB `role_name` 도 `ROLE_ADMIN`/`ROLE_USER`).
- **비밀번호**: `PasswordPolicy`(min-length·문자종류 검증) + `BcryptEnforcingPasswordEncoder`(신규 인코딩=BCrypt, `allow-noop` 토글). 회원가입/본인변경/관리자초기화 플로우에 정책 연결됨. 저장값은 `{bcrypt}...`(또는 로컬 시드 `{noop}...`).
- **로그인 시도 제한**: `LoginAttemptService` — in-memory(기본, 단일 인스턴스) / redis(다중 인스턴스 공유). 임계치 초과 시 `429 LOGIN_LOCKED`. 실패 카운트 키 정책 `key-strategy`: `login-id`(기본) | `login-id-and-ip`(IP 는 `ClientIpResolver` 가 `X-Forwarded-For`→`getRemoteAddr()` 폴백으로 추출, **XFF 위조 가능**이라 신뢰 프록시 환경 한정).
- **비밀번호 변경 인가 분리** (user-service):
  - `PATCH /api/v1/users/me/password` — 본인만(현재 비번 필요, 컨텍스트 사용자 해석)
  - `PATCH /api/v1/users/{id}/password/reset` — 관리자 강제 초기화(`@PreAuthorize("hasRole('ADMIN')")`, 현재 비번 불요)
- **비밀번호 만료/이력**(ISMS-P, framework-security 확장): `PasswordLifecycleService` — 직전 N개 재사용 금지·변경주기 만료 판정(`framework.security.password.{expiry,history}`). 기능 off 면 no-op, 이력 없는 레거시 사용자는 강제 만료 안 함. 업무코드가 비번 변경 시점에 호출.
- **동시(중복) 로그인 제어**(framework-security 확장): `ConcurrentSessionService` — 사용자당 세션 수 제한, 초과 시 `EVICT_OLDEST`(기존 토큰 블랙리스트) 또는 `REJECT`(409). `framework.security.concurrent-session.*`, store=memory|jdbc. 기본 off.
- **감사/접속 로그**(framework-audit, 선택): `@AuditLog` 메서드 + 로그인 성공/실패/로그아웃 이벤트를 표준 적재. security→audit 는 이벤트(`LoginAuditEvent`) 단방향(순환 회피). store=logging|jdbc(jdbc 일 때 `GET /api/v1/audit/logs`).
- **웹 보안 필터**(framework-secure-web, 선택): 보안헤더·경로조작·인젝션 스크리닝·CSRF 더블서브밋. 필터 계층(디스패처 이전)이라 거부 응답은 `SecureWebResponder` 가 표준 JSON 직접 기록. CSRF 는 Spring Security(csrf disable)와 독립.
- **파일 업로드 보안**(framework-file, 선택): 저장명 UUID 생성·경로 정규화(base-path 이탈 차단)·위험 확장자 차단·크기 제한에 더해 (옵트인) **콘텐츠 타입 검증**(`validation.content-type-detection`=Tika 매직넘버로 실제 MIME 판정 → png 위장 jsp/exe 등 차단, 검출 MIME 을 메타에 기록)·**확장자↔MIME 계열 정합**(`validation.enforce-extension-match`=선언 .png 인데 실제 PDF/zip/실행파일이면 거부, 컨테이너 계열은 통과)·**안티바이러스 스캔**(`scan.enabled`+`scan.type=clamav`=ClamAV INSTREAM 게이트, 저장 전 검사, 감염 거부, fail-closed 기본)·**저장소 at-rest 암호화**(`storage.encrypt`=AES-CBC 스트리밍, 코어 `AesCryptoService` 키 재사용, local/nas/s3 일괄). 전부 기본 off=기존 동작. 다운로드는 (Range 지원 저장소면) **HTTP Range 206 스트리밍**, S3 는 **presigned PUT/GET** 발급으로 대용량 직행.
- **소셜 로그인 / OIDC**(framework-oauth-client, 선택): 외부 IdP(google/kakao/naver) OAuth2/OIDC 인가코드 흐름으로 신원확인 → `OAuthUserResolver`(앱 구현) 가 외부신원→`AuthenticatedUser` 매핑(JIT 가입) → security 의 JwtProvider/TokenStore 로 **자체 JWT 발급**(외부 토큰은 신원확인용일 뿐, 이후 자체 토큰으로 운영). state(CSRF) memory|redis(다중파드 redis). 엔드포인트 `/api/v1/auth/oauth/{provider}/{authorize|callback}`(이미 `/api/*/auth/**` permitAll). 발급기 `OAuthTokenIssuer` 교체로 LoginService(동시로그인/감사) 통합 가능. **OAuth 클라이언트(소비)이며 인증서버가 아님.** **OIDC 강화(2026-06-03, per-provider `oidc.enabled` 기본 off)**: 켜면 callback 에서 **id_token 을 검증**(JWKS 의 RSA/EC 서명 또는 HS=client-secret HMAC · iss · aud⊇client-id · exp/nbf±skew · nonce · sub) + **discovery 자동적용**(issuer/discovery-uri→엔드포인트 보충, 지연·1회·캐시) + **nonce 바인딩**(authorize↔callback, state 와 함께 저장). 신원은 검증된 id_token 클레임으로 구성(userInfoUri 있으면 빈 필드만 userinfo 보충). 비OIDC(kakao/naver)는 기존 흐름 불변. id_token 검증은 jjwt 재사용(security `api` 전이). 사용법 `docs/modules/OAUTH_CLIENT.md`·`docs/modules/OIDC_HARDENING.md`.
- **게이트웨이 엣지 인증**(services/gateway, 선택): 게이트웨이(WebFlux)에서 access JWT 를 1차 검증(서명+만료+typ) 후 신뢰 헤더 `X-User-Id`/`X-User-Roles` 주입 — 클라이언트가 보낸 동일 헤더는 **항상 먼저 제거**(스푸핑 차단), 다운스트림 `framework-context` 가 그대로 사용. security(서블릿)와 WebFlux 충돌 회피 위해 **jjwt 만 직접 의존**(framework-security 비의존), secret 은 `framework.security.jwt.secret` 과 동일 값을 `JWT_SECRET` 으로 공유. 검증한 userId 로 레이트리밋이 사용자 단위로 동작. `gateway.auth.enabled` 기본 off. **중앙 로그아웃(SSO A, 2026-06-03)**: 검증 후 jti(`Claims.getId()`)를 `RedisGatewayTokenBlacklist.isBlacklisted("bl:"+jti)`(reactive)로 조회해 **로그아웃된 토큰을 엣지에서 401 차단**(키 prefix 는 `RedisTokenStore` 와 동일). 토글 `gateway.auth.blacklist-check.enabled`(기본 off, redis 전용·미설정 시 fail-fast). 사용법 `docs/modules/GATEWAY_EDGE_AUTH.md`·`docs/modules/SSO_CENTRAL_LOGOUT.md`.
- **중앙 로그아웃 / logout-all**(framework-security, SSO A, 2026-06-03): `LoginService.logout` 이 access jti 블랙리스트 + refresh 제거 + (선택) 동시세션 해제 → 게이트웨이 blacklist-check 가 읽으면 전 서비스 즉시 무효. **`LoginService.logoutAll(access, ip)`** = 토큰→userId 검증 후 **현재 토큰 항상 블랙리스트**(레지스트리 없어도 안전망) + `ConcurrentSessionService.activeSessions(userId)` 순회로 각 refresh 제거 + accessJti 블랙리스트 + unregister. `POST /api/v1/auth/logout-all`. 완전한 "모든 기기" 커버는 `framework.security.concurrent-session.enabled=true` + 공유 `token.store=redis` 전제. 감사는 `LOGOUT` 타입 재사용(detail `logout-all:N`).
- **SAML 2.0 SP / 표준 프로토콜 SSO**(framework-saml-sp, SSO B-SAML, 선택, 2026-06-04): SAML 만 말하는 외부/사내 IdP(공공 통합인증·일부 그룹 SSO·Keycloak/Azure AD SAML 모드)에 **SP 로 연동** — Spring Security SAML2 SP 가 IdP 메타데이터로 RelyingParty 등록·AuthnRequest 발행·**서명된 assertion 검증**까지 처리하고, ACS 성공 시 **서버 세션 없이** NameID/속성 → `SamlAttributeMapper` 정규화(`SamlUserInfo`) → `SamlUserResolver`(앱 구현, `OAuthUserResolver`/`Authenticator` 대칭) → `SamlTokenIssuer`(security `JwtProvider`/`TokenStore` 재사용)로 **자체 JWT 발급**(수기 JSON, 필터 계층이라 Jackson 비의존). 외부 신원은 신원확인용일 뿐 이후 운영은 기존 JWT/게이트웨이/context 그대로(OAuth/OIDC 와 같은 결). 전용 `SecurityFilterChain`(`securityMatcher("/saml2/**","/login/saml2/**")` + 높은 우선순위 + `@AutoConfiguration(after=SecurityAutoConfiguration)`)으로 메인 인증 체인 무수정. 3단 토글=모듈+`framework.saml-sp.enabled`+`SamlUserResolver` 빈+`registrations.<id>.metadata-uri`. 엔드포인트=SS 기본(`/saml2/service-provider-metadata/{id}`·`/saml2/authenticate/{id}`·`/login/saml2/sso/{id}`). ⚠️ OpenSAML 전이=첫 "새 외부 의존성 0" 예외(Central 밖→Shibboleth repo). 멀티 파드는 현재 게이트웨이 스티키 세션(redis AuthnRequest 저장소는 후속). 사용법 `docs/modules/SAML_SP.md`, 설계 `docs/NEXT_SSO.md` §5.
- **dev-auth**: 개발 초기 토큰 없이 권한만 바꿔 호출하는 우회 모드(`local,dev` 프로파일). 운영 비활성.

## 5. 환경 / 배포
- **운영 프로파일**: PostgreSQL 드라이버 + Flyway(`db/migration` 만, `db/seed-local` 제외). 비번 `allow-noop:false`(평문 차단).
- **컨테이너**: `deploy/docker/Dockerfile` — jar 를 CI(Jenkins)가 먼저 `bootJar` → `JAR_FILE` 로 주입, 레이어 추출 후 엔트리포인트 `org.springframework.boot.loader.launch.JarLauncher`. 비루트 실행.
- **k8s**: `deploy/k8s/*` — `user-service replicas: 2`, actuator 프로브(`/actuator/health/liveness|readiness`), 설정은 configmap/secret.
- **다중 인스턴스(중요)**: replicas≥2 이므로 운영은 `login-attempt.type=redis` + `token-store.type=redis` 필수. in-memory 는 인스턴스별이라 잠금/세션 공유 안 됨(잠금 우회 가능).
- **CI/CD**: `deploy/cicd/Jenkinsfile` 한 곳에서 빌드·테스트·게이트(spotlessCheck/jacoco/dependencyCheck/sonar)·이미지·롤아웃.

## 6. 컨벤션 & 함정 (되돌리지 말 것)
- enum `ErrorCode.Common` 상수 추가 시 종결 `;` 위치 주의(직전 줄을 `,` 로).
- 프레임워크 빈은 `@AutoConfiguration` + `META-INF/.../AutoConfiguration.imports` 로 등록. 신규 모듈도 동일.
- `@PreAuthorize` 는 `@EnableMethodSecurity`(SecurityAutoConfiguration)로 전역 적용. authority 에 `ROLE_` 접두사 있으므로 `hasRole('ADMIN')`(접두사 중복 X).
- `BcryptEnforcingPasswordEncoder`: `encode()` 는 `{bcrypt}` 접두사 포함 저장, `matches()` 가 접두사로 알고리즘 식별 → 저장값에서 접두사 임의 제거 금지. `allow-noop=false` 면 `{noop}` 매칭 거부(예외 아님, false).
- 로그인 잠금: in-memory 는 임계치 전 카운터 만료 없음 / redis 는 `la:fail` 에 lock-duration 롤링 윈도우 TTL(의도적 차이). 분산 환경은 redis.
- IP 키: `X-Forwarded-For` 위조 가능 → 신뢰 프록시가 헤더 세팅하는 환경에서만 `login-id-and-ip` 의미. 외부 직접 노출 시 헤더 신뢰 끄기.
- **API 파괴적 변경 이력**: (a) 회원가입 `password` 필수가 됨(이전 누락) — 비번 없는 바디는 400. (b) 구 `PATCH /users/{id}/password` 제거 → `/me/password` + `/{id}/password/reset` 로 분리.
- `LoginService.login(command)` 단일인자 시그니처 유지(하위호환). IP 결합은 `login(command, clientIp)`.
- **신규 모듈 등록 필수**: 모듈 폴더를 추가하면 반드시 `settings.gradle` 에 `include 'framework:framework-<X>'` 를 넣는다. 누락 시 `project '<X>' not found in project ':framework'`. 드롭인 배포는 모듈 폴더 + 완성 `settings.gradle` 을 한 zip 으로.
- **Boot 4 모듈 분리 주의**: `org.springframework.boot.http.client.*`(ClientHttpRequestFactoryBuilder/Settings)는 `spring-boot-starter-web` 컴파일 경로로 안 딸려온다 → RestClient 타임아웃은 spring-web 의 `SimpleClientHttpRequestFactory` 로 설정. `RestTemplateBuilder` 도 `org.springframework.boot.restclient` 로 이동.

  - **암호화(AES-CBC) 저장소는 Range 미지원**: IV‖암호문 CBC 는 임의 오프셋 복호화 불가 → `EncryptingFileStorage` 가 `RangeReadableStorage` 를 **구현하지 않는다** → BPP 가 RangeReadable 한 delegate 를 감싸면 capability 가 사라져 컨트롤러가 전체 다운로드로 폴백(올바른 동작). Range 와 at-rest 암호화는 양립 불가임을 전제.
  - **presigned PUT 업로드는 서버 본문 비경유 → 콘텐츠/AV 검증 불가**: `registerExternalUpload`(presigned-complete)는 **파일명 확장자 화이트리스트만** 강제하고 매직넘버/스캔은 못 한다. 신뢰경계 밖 업로드면 비동기 후처리 스캔을 별도로. (서버 멀티파트 경유 업로드 `POST /api/v1/files` 는 전 게이트 적용.)
  - **AV 는 fail-closed 기본**: `ClamavFileScanner` 는 데몬 접속/통신 실패 시 `failOpen=false`(기본)면 `BusinessException` 으로 **거부**(ISMS-P 보안 우선). 가용성 우선이면 `scan.fail-open=true`. INSTREAM 은 **순수 JDK 소켓**(`zINSTREAM\0`→[len(4B BE)][chunk]…[0]→`stream: OK`/`<sig> FOUND`) 이라 **새 외부 의존성 0**. 단위검증은 mock `ServerSocket` 으로 청크 프레이밍 왕복.
  - **확장자↔MIME 은 "계열" 정합만**: tika-core 매직넘버는 컨테이너(zip/OLE2)까지만 정확 → docx↔xlsx 미세구분 불가. `ExtensionContentTypePolicy` 는 카테고리(이미지/PDF/텍스트/OOXML/OLE2) 단위 허용 집합으로 **명백한 위장만** 차단(오탐 방지). 규칙 없는 확장자는 1차 화이트리스트가 거른 뒤이므로 통과.

  - **오토컨피그에 빈 추가 시 기존 deep-stub mock 테스트 회귀(2026-06-03)**: `S3FileStorageAutoConfigurationTest` 는 `FileStorageProperties` 를 `Answers.RETURNS_DEEP_STUBS` mock 으로 넣는다 → 새로 추가한 `s3Presigner` 빈이 `Region.of(props…getRegion())` 를 **실제로 실행**하는데 mock `getRegion()`=null 이라 `Region.of(null)` → 컨텍스트 기동 실패(`hasNotFailed()` AssertionError). 기존 `s3Client` 빈은 mock 제공으로 `@ConditionalOnMissingBean` 양보돼 region 미접근이었음. **해소: 새 빈 타입(`S3Presigner`)도 `s3Client` 처럼 mock 을 `withBean` 으로 제공**(양보 유도). 교훈: 기존 컨텍스트 러너 테스트가 deep-stub mock 을 쓰면, 오토컨피그에 **새 @Bean 을 추가할 때 그 빈이 참조하는 외부 타입 mock 도 함께 제공**해야 회귀가 없다(운영 코드는 프로퍼티 기본값으로 정상).
- **분산 캐시는 core 보다 먼저 떠야 한다(framework-cache-redis, 2026-06-03)**: core `CacheAutoConfiguration` 이 Caffeine `CacheManager` 를 `@ConditionalOnMissingBean(CacheManager)+matchIfMissing=true` 로 **항상** 등록 → Boot 네이티브 `spring.cache.type=redis`(또는 평범한 `@AutoConfiguration`)는 이미 빈이 있어 **백오프되어 무력**. 그래서 `RedisCacheAutoConfiguration` 은 **`@AutoConfiguration(before=CacheAutoConfiguration.class)`** 로 Redis `CacheManager` 를 먼저 올려 core 가 자기 조건으로 물러나게 한다(datasource Registrar 의 before-순서 교훈과 같은 결). 끄면(기본) core Caffeine 그대로. `@EnableCaching` 재선언 금지(core 가 이미 켬).
- **Redis 캐시 직렬화는 JDK(`RedisSerializer.java()`), Jackson2 금지(framework-cache-redis)**: Spring Data Redis 의 `GenericJackson2JsonRedisSerializer`/`Jackson2JsonRedisSerializer` 는 **Jackson 2(`com.fasterxml.jackson.databind`)** 라 본 스택(Jackson 3) 규약 위반(클래스패스 부재 → 컴파일/런타임 실패) → 쓰지 않는다. 레포 전역 일관(`RedisTokenStore`/MFA 챌린지도 Redis 에 Jackson 미사용, StringRedisTemplate+수기). 캐시 값은 `Serializable` 필요. JSON 직렬화가 꼭 필요하면 **앱이 `RedisCacheConfiguration` 빈을 직접 등록**(본 모듈 기본은 `@ConditionalOnMissingBean(RedisCacheConfiguration)` 라 앱 빈이 우선) — 그 안에서 Jackson 3 `JsonMapper` 기반 직렬화기를 앱 책임으로 구성. data-redis 는 `compileOnly`+`testImplementation`(introspection/테스트가 `RedisConnectionFactory`/`RedisCacheManager` 참조).
- **Logback 컨버터는 Spring DI 를 못 받는다(framework-log-masking, 2026-06-03)**: `conversionRule` 로 등록되는 `MessageConverter`(`MaskingMessageConverter`)는 **Logback 이 직접 인스턴스화**(컨테이너 밖, 필터·EPP 처럼 DI 불가) → Spring 이 조립한 `SensitiveDataMasker` 를 **정적 다리(`MaskingSupport`, volatile 단일 참조)** 에 라이프사이클 빈(`LogMaskingInstaller` InitializingBean/DisposableBean)이 설치/해제한다. **미설치 시(부팅 초기 로그·순수 Logback 사용)에도 `withDefaults()` 로 폴백** → 원문 PII 노출 0(설치 전이라고 마스킹이 꺼지지 않음). 정적 상태라 테스트는 `@AfterEach MaskingSupport.clear()` 로 격리. `MaskingSupport` 의 정적 필드는 `@Autowired` 가 아니므로 ArchUnit 필드주입 규칙과 무관.
- **`%msg`/`%message` 재정의 금지 → 별도 단어 `%mmsg`(framework-log-masking)**: Logback 에서 표준 conversionWord 를 재정의하면 경고가 날 수 있어, 마스킹 패턴은 표준 `%msg` 를 건드리지 않고 **새 단어 `%mmsg`** 를 둔다(`logback-masking.xml` 의 `<conversionRule conversionWord="mmsg" …>`). 컨버터는 `super.convert(event)`(인자 치환 후 최종 텍스트)를 마스킹하므로 `{}` 인자로 흘러든 PII 도 잡힌다.
- **로그 마스킹 오탐(false positive) 정책(framework-log-masking)**: 자유 로그에서 임의 숫자열을 과하게 가리면 운영 가독성이 떨어진다 → 정규식 경계를 엄격히(`(?<![0-9])…(?![0-9])`, 카드=16자리 또는 4-4-4-4 그룹 한정). **계좌번호는 일반 숫자열과 충돌이 잦아 기본 off**(`rules.account=true` 로만). 규칙 적용 순서는 **자릿수가 긴 카드를 먼저**(card→rrn→phone→email→account→custom) 두어 상호 오삼킴 방지. 커스텀 패턴은 매칭 전체를 별표(상한 64자) 마스킹.
- **계좌 규칙은 dash-grouped 숫자열을 광범위하게 잡는다(framework-log-masking, 2026-06-03 테스트 교훈)**: 계좌 정규식 `(?<![0-9])\d{2,6}-\d{2,6}-\d{2,6}(?![0-9])` 은 구분자(`-`)로 묶인 2~6자리 3그룹이면 **휴대폰 `010-1234-5678` 같은 비계좌 숫자열도 매칭**한다(이것이 계좌를 기본 off 로 둔 이유 — 켤 땐 환경의 숫자열 포맷이 계좌와 충돌하지 않는지 점검). **테스트 작성 시 주의**: `account=true` 를 켠 채 다른 규칙(예: phone off)이 "안 가린다"를 검증하려면, 계좌 패턴에도 안 걸리는 **구분자 없는 입력**(예: `01012345678`)으로 격리해야 한다 — 안 그러면 계좌 규칙이 잡아 오탐 단언이 깨진다(`LogMaskingAutoConfigurationTest#appliesRuleTogglesAndCustomPatterns` 1차 실패 원인. **모듈 코드 정상, 테스트 결함**이었고 입력을 dashless 로 교체해 해소).
- **Boot 구조화 로깅엔 `%mmsg` 컨버터 미적용(framework-log-masking↔observability)**: observability 의 JSON 구조화 로그(`logging.structured.format.*`)는 `PatternLayout` 을 우회하므로 `%mmsg` 컨버터가 동작하지 않는다 → 그 경로의 PII 는 **1차(SensitiveDataMasker 빈 명시 호출)** 로 가려야 한다(README 명시). 정규식 기반이라 완전 탐지는 보장 못 함(방어망 성격) — 민감 지점은 1차로 확실히.
- **요청 컨텍스트는 상속형 ThreadLocal 을 쓰지 않는다(framework-context, 2026-06-03)**: `ContextHolder` 는 평범한 `ThreadLocal` — 가상 스레드(core 의 `Executors.newVirtualThreadPerTaskExecutor`)/풀에서 `InheritableThreadLocal` 은 컨텍스트 누수를 일으키므로 금지. 다른 스레드 전파는 **항상 명시적**: `@Async`/풀=`ContextTaskDecorator`(제출 스냅샷→작업 스레드 복원→종료 후 원복, 풀 재사용 누수 방지), 아웃바운드=`ContextPropagationInterceptor`(헤더). `ContextBindingFilter` 는 요청 종료(예외 포함) 시 `ContextHolder.clear()`+자기 MDC 키 제거. 정적 상태라 테스트는 `@AfterEach ContextHolder.clear()`+`MDC.clear()` 로 격리.
- **테넌트 필수 검증은 필터가 아니라 서비스 레이어에서(framework-context)**: 필터는 디스패처 이전이라 `GlobalExceptionHandler` 밖(secure-web `SecureWebResponder` 와 같은 결) → 필터에서 거부 JSON 수기작성 대신 `ContextBindingFilter` 는 **바인딩만** 하고, 테넌트 필수 여부는 서비스/컨트롤러에서 `ContextHolder.get().hasTenant()` 검사 후 `BusinessException`(표준 에러처리 경유)으로 던진다. `HeaderContextResolver` 는 신뢰 헤더를 읽기만 함 — 외부 경계 위조 방지는 게이트웨이/인증 계층 책임.
- **컨텍스트 필터 순서 = MdcTraceFilter 바로 안쪽(framework-context)**: `MdcTraceFilter`=`HIGHEST_PRECEDENCE`(최외곽, traceId MDC + finally `MDC.clear()`), `ContextBindingFilter`=`HIGHEST_PRECEDENCE+10`(그 안쪽에서 tenantId/userId 를 MDC 에 덧붙임, traceId 가 이미 있는 상태). 트레이스 필터의 외곽 `MDC.clear()` 와 충돌 없음.
- **메타(EXIF/GPS) 제거는 별도 strip 단계가 아니라 리인코딩 부수효과(framework-image, 2026-06-03)**: `DefaultImageProcessor.process()` 는 디코드→`BufferedImage`→ImageIO 리인코딩이라 **원본 메타데이터가 출력에 보존되지 않는다**(GPS 포함 자동 제거). 즉 `process()`/`thumbnail()` 출력은 **항상 메타 없음** — 원본 메타를 보존해야 하면 이 모듈을 거치지 말고 별도 경로로. orientation 만 픽셀에 굽고(burn-in) 나머지 메타는 버린다.
- **EXIF orientation 보정은 best-effort(framework-image)**: `ExifOrientation.read(bytes)` 는 메타 라이브러리 없이 JPEG APP1 → TIFF 헤더(II/MM 엔디안)→IFD0 → 태그 0x0112 를 직접 파싱하되, **APP1 부재·비JPEG·잘린 바이트·범위밖 값·널이면 예외 없이 `NORMAL`(1) 반환**(원본 방향 유지). PNG 등은 orientation 개념이 없어 항상 NORMAL. 따라서 회전 보정은 "되면 좋고 안 되면 원본" 성격 — 깨진 EXIF 로 처리 전체가 실패하지 않는다. (secure-web 의 수기 파싱과 같은 결: 신뢰 못 할 입력은 안전하게 무시.)

- **JPEG 출력은 알파를 흰 배경으로 평탄화(framework-image)**: JPEG 은 투명도를 표현 못 하므로 알파 채널 이미지를 JPEG 으로 인코딩할 때 흰 배경 위에 합성한다(검정 박스 방지). **투명 보존이 필요하면 출력 포맷을 PNG 로** 지정. 출력 포맷 화이트리스트는 JPEG/PNG 뿐(그 외 요청은 `UNSUPPORTED_FORMAT`).

- **요청 검증 — `@Valid 바디` vs `메서드 파라미터`는 다른 예외(2026-06-03)**: `@Valid @RequestBody` 실패는 `MethodArgumentNotValidException`(기존 처리), 컨트롤러 **메서드 파라미터**(`@RequestParam`/`@PathVariable` + 제약) 실패는 Spring Framework 6.1+/7 의 **`HandlerMethodValidationException`**(`org.springframework.web.method.annotation`, `ResponseStatusException` 상속). 후자를 안 잡으면 표준 `ApiResponse` 가 아닌 기본 ProblemDetail 로 응답돼 형식이 깨진다 → `GlobalExceptionHandler` 에 핸들러 추가(400 INVALID_INPUT 통일). 미인증 진입점 로그인(`LoginCommand`)에 `@NotBlank`+`@Valid` 로 빈 자격증명 차단. validation 은 framework-core 가 `api 'spring-boot-starter-validation'` 으로 전이 노출 → 서비스는 별도 의존 불필요.
- **spotless = Palantir Java Format(google 아님) + 설정 캐시 충돌 주의(2026-06-03)**: 포맷터는 `palantirJavaFormat()`. `org.gradle.configuration-cache=true` + spotless 기본 줄바꿈정책 `GIT_ATTRIBUTES` 가 `lineEndingsPolicy`(직렬화 불가 provider)로 `:spotlessGroovyGradle` 등에서 빌드를 깬다(gradle#19113/spotless#987) → **`lineEndings = com.diffplug.spotless.LineEnding.UNIX`** 로 고정(Windows 개발+Linux CI 일관성=LF). spotless 를 `subprojects` 뿐 아니라 **루트에도 적용**해야 루트 `build.gradle`/yaml/sql/md 가 검사된다. YAML/SQL/Gradle 은 의미 보존 위해 공격적 재포맷터(jackson/dbeaver/greclipse) 대신 화이트스페이스 위생만. `encoding 'UTF-8'`(한글 주석 보호) 필수. 에러 후엔 `.gradle/configuration-cache` 비우고 재실행. (`docs/SPOTLESS_NOTES.md`)
- **설정값 암호화 EPP 완료(2026-06-03, 설계 `docs/NEXT_YAML_PASSWORD_ENCRYPTION.md`)**: `ENC(...)` 프로퍼티 복호화는 `framework-core` 의 **`EncryptedPropertyEnvironmentPostProcessor`**(Boot4 패키지 `org.springframework.boot.EnvironmentPostProcessor` + framework-core 신설 `META-INF/spring.factories` 키 정확히 — `.imports` 아님)가 한다. EPP 는 컨텍스트 전이라 빈 사용 불가 → `AesCryptoService`(AES-GCM 문자열) **직접 생성**. 마스터키=`framework.crypto.aes-secret`/`AES_SECRET` 는 **평문 주입**(`ENC()` 불가, 닭-달걀 — 코드가 마스터키가 ENC면 기동 실패시킴). **복호화는 지연(lazy)**: 각 `EnumerablePropertySource` 를 `DecryptingPropertySource` 로 교체, 값 읽는 시점에만 복호화 → 프로파일별 yaml 늦게 들어와도 누락 없음. **함정**: ① enumerable 래퍼는 `getPropertyNames()` 도 위임해야 Binder 가 ENC 값을 봄(누락 시 바인딩 실패). ② `getOrder=LOWEST_PRECEDENCE`(ConfigData 적재 이후 래핑). ③ 토글 `framework.crypto.config-decryption.enabled` **기본 on**(일반 규약은 off지만 ENC 없으면 완전 무동작이라 on 이 안전 — off면 토글 잊었을 때 `ENC(...)` 리터럴이 그대로 흘러 조용히 깨짐). ④ 사전 스캔으로 ENC 가 하나도 없으면 마스터키 없이도 무동작(통과). ENC 있는데 마스터키 없으면 명확히 기동 실패. ⑤ 복호화 실패=기동 실패(GCM 인증, 깨진 값 미통과), 예외 메시지/로그에 평문·키 노출 금지. ⑥ prod 마스터키 가드 `AesMasterKeySafetyGuard`(JWT 가드 패턴, `CryptoAutoConfiguration` 빈) 동봉. ⑦ 토큰 생성 CLI=`CryptoCli`(`AES_SECRET` env, `encrypt <평문>` → `ENC(...)`). Jackson 무관·신규 의존성 0. (운영 정책=권장 (a): prod 비밀은 env/시크릿 주입 유지, ENC 는 dev 편의 + 저장소 평문 미잔존.)
- **가상스레드 executor 의 close()가 곧 join(framework-file-batch, 2026-06-03)**: `FileBatchProcessor` 는 `try (var exec = Executors.newVirtualThreadPerTaskExecutor()) { ... submit ... }` 구조 — `ExecutorService.close()`(AutoCloseable, Java 19+)가 **제출된 모든 작업이 끝날 때까지 블록(join)** 한다. 따라서 `future.get()` 결과 수집은 **try 블록 밖(close 이후)** 에서 한다(이미 다 끝난 상태라 블록 안 됨). 동시도는 가상스레드 무제한이 아니라 `Semaphore(parallelism)` 로 제한(디스크/스토리지 보호). fail-fast 는 `AtomicBoolean abort` — 실패 시 set, 후속 작업은 `processOne` 진입 시 확인해 SKIPPED.
- **교차 아이템 검증은 apply 가 아니라 BatchPreflight 에서(framework-file-batch)**: 개별 `BatchFileOperation.apply(item)` 은 **다른 아이템을 못 본다** → 이름 충돌 같은 전체를 함께 봐야 하는 검증은 `BatchPreflight` capability(ISP 확장)로 분리. `FileBatchProcessor` 가 IO 이전에 1회 `preflight(전체)` 호출(실패 시 부분 손상 없이 전체 중단). `RenameOperation` 이 이를 구현해 충돌을 미리 확정(`resolved` 맵)하고 `apply` 는 읽기만 → 병렬 안전. **연번은 공유 카운터가 아니라 입력 index 기반**이라 병렬에서도 결정적(같은 입력=같은 출력).
- **@ConditionalOnClass 백오프는 중첩 @Configuration 에 — 테스트는 FilteredClassLoader(framework-file-batch)**: image/archive 위임은 `compileOnly`(전이 노출 금지)이고, op 팩토리 빈은 `FileBatchAutoConfiguration` 의 **중첩 static @Configuration**(`ImageOpsConfiguration`/`ArchiveOpsConfiguration`)에 `@ConditionalOnClass(ImageProcessor/Archiver)`+`@ConditionalOnBean` 을 달아 Spring ASM 이 **클래스 로딩 없이** 평가하게 한다(introspection 함정 회피, image 모듈 선례와 동일). 테스트는 image/archive 를 `testImplementation` 재선언해 위임/백오프를 실제로 검증하고, **클래스 부재 백오프**는 `org.springframework.boot.test.context.FilteredClassLoader` 로 시뮬레이션 — ⚠️ 단 **필터한 타입을 참조하는 사용자 설정은 그 컨텍스트에서 로드 불가**하므로, image 를 가린 테스트엔 Archiver 빈만(ImageProcessor 미참조), archive 를 가린 테스트엔 ImageProcessor 빈만 주는 단일-위임 설정으로 분리해야 한다(공용 DelegateBeans 는 양쪽 참조라 필터 테스트에서 깨짐).
- **얇은 위임 — 일괄처리는 인코딩을 직접 안 한다(framework-file-batch)**: 변환=image, 압축=archive 에 **위임**만 한다(새 인코딩 코드 0). `CompressOperation` 은 SPI 가 per-item `apply` 라 **파일별 gzip**(`<name>.gz`)으로 구현 — "여러 파일→1개 zip" 같은 **집계** 압축은 일괄처리가 아니라 `Archiver.zip(entries, out)` 단일 호출이 맞다(설계서 모호성 해소). 출력은 sourcePath 가 있으면 같은 디렉토리에 sibling 으로 스트리밍 기록, 없으면 결과 바이트를 본문(`BatchContent`)으로 반환 — outputDir 파라미터는 두지 않음(작업이 자체 IO). 결과 이름은 항상 `BatchSafety.requireSimpleName`(경로구분자/`.`/`..`/드라이브 거부)로 단일 파일명 강제.
- **SFTP 는 연결 상태형 — S3 무상태 모델과 다르다(framework-file-sftp, 2026-06-03)**: ① `SshClient` 는 1회 `setUpDefaultClient().start()` 후 **재사용**(start 는 NIO 만 띄울 뿐 연결 아님 → 더미 host 로도 빈 생성 OK, 연결은 작업 시점). 빈은 `AutoCloseable` → 컨텍스트 종료 시 `client.stop()`. ② **작업마다 세션을 새로 열고 닫는다**(풀 없음 — stale 연결/스레드안전 버그 회피, SI 볼륨엔 충분). ③ **`load`/`loadRange` 가 돌려주는 스트림은 세션을 물고 있어야** 한다 → `withSftp`(try-with-resources)로 닫으면 안 됨. `SessionBoundInputStream`(FilterInputStream)이 스트림 close 시 `sftp`→`session` 을 함께 정리. 호출자는 반드시 스트림을 닫아야 세션 누수가 없다. ④ **호스트키 검증 기본 strict**(`KnownHostsServerKeyVerifier(RejectAll, known_hosts)` = fail-closed); `strict-host-key-checking=false` 면 `AcceptAllServerKeyVerifier`+경고(개발 전용). ⑤ **loadRange 는 handle-offset 대신 skip+bounded**(open InputStream → `skipFully(start)` → `BoundedInputStream(len)`) — skip()이 0 반환 시 read 폐기 폴백. ⑥ `mkdir -p` 는 `ancestorDirs` 로 조상까지 stat-후-mkdir(`ALREADY_EXISTS` 무시), 없는 파일 delete 는 `SSH_FX_NO_SUCH_FILE` 무시(멱등). ⑦ MINA 타입은 `implementation`(전이 금지, S3 와 달리 SshClient 빈 미노출 — 깊은 커스터마이즈는 앱이 `FileStorage` 빈 직접 정의). ⑧ **sshd-core/sshd-sftp 동일 버전, 3.0.0 은 마일스톤(2.x API 비호환) → 2.16.0 고정**(BOM 밖, 카탈로그 sshd + 루트 ext sshdVersion). ⑨ 라운드트립 테스트는 **내장 MINA SshServer**(setUpDefaultServer+SimpleGeneratorHostKeyProvider+PasswordAuthenticator+SftpSubsystemFactory+VirtualFileSystemFactory(@TempDir)) — `setSubsystemFactories` 는 `List<SubsystemFactory>` 불변이라 명시적 타입 리스트 사용. ⚠️ 작성환경 Maven 차단 → MINA 컴파일은 받는 쪽(순수 경로/Range 헬퍼는 JDK 단독 22/22 검증).
- **블록 주석 안에 `*/` 금지(2026-06-03, OIDC `IdTokenVerifier` 에서 발생)**: javadoc/블록 주석 텍스트에 `RS*/ES*/PS*` 처럼 `*/` 시퀀스가 들어가면 **주석이 그 지점에서 닫혀** 뒤 문장이 코드로 파싱돼 `error: class, interface, enum, or record expected`(+spotless 가 엉뚱한 `toggle(toggle)` lint 로 오인). 알고리즘 나열은 `RS/ES/PS 계열`처럼 `*/` 를 피한다(`HS*` 는 `*` 단독이라 무해). brace/paren·forbidden-import 정적 점검으로는 **안 잡힘** → 점검 항목에 "주석 내 `*/`"(`grep -nE '\*/[^ ]'`) 추가.
- **JWKS 키 회전 재조회는 신선도 가드를 우회해야(2026-06-03, `JwksKeyResolver`)**: `refresh()` 의 "캐시가 신선하면 그대로 반환" 재진입 가드(동시 갱신 dedupe 용)가 **unknown kid 강제 재조회까지 차단**해 fetch 가 1회만 일어났다(회전 테스트 fetch=2 기대 실패) → **lazy/forced 분리**(forced 는 가드 우회). 단 가짜 kid 남용으로 인한 JWKS 조회 폭주 방지로 **강제 재조회는 per-jwks_uri 쿨다운(60s) throttle**(`lastForcedRefetch` 기본 EPOCH → 최초 unknown kid 는 즉시 허용, 직후 동일 kid 재요청은 생략). kid 없고 키 1개면 그 키 사용(단일키 관용).
- **OIDC nonce 는 state 와 함께 1회용 저장(2026-06-03)**: `OAuthStateStore` 에 `save(state,providerId,nonce,ttl)`/`consumeState():Optional<StateData(providerId,nonce)>` **default 메서드 추가**(기존 `save(...,ttl)`/`consume():Optional<String>` 보존=하위호환). Redis 값은 `providerId\nnonce`(개행은 providerId 에 없음, 비OIDC 는 nonce 없이 providerId 만). OIDC 경로만 nonce 버전 사용, 비OIDC 는 기존 그대로. id_token 검증은 **jjwt 재사용**(jjwt-api 가 framework-security `api` 로 oauth-client 컴파일 클래스패스에 전이 — 별도 추가 불필요, 테스트 서명/검증용 impl/jackson 만 `testRuntimeOnly`).
- **OpenSAML = 첫 비-Central 저장소 의존(2026-06-04, framework-saml-sp)**: ⚠️ **OpenSAML 4+ 는 Maven Central 에 게시되지 않는다**(라이선스/면책 — 실측 확인). `spring-security-saml2-service-provider` 가 OpenSAML 을 **전이로** 끌어오고 **버전도 Spring Security 가 관리**하므로 (a) opensaml 을 **명시 선언/핀하지 말 것**(SS 관리 버전과 어긋남 + 루트 ext 미정의 시 설정 단계 실패 — 초안의 `${openSamlVersion}` 가 정확히 이 함정이었음, 제거함), (b) 루트 `allprojects.repositories` 에 **`https://build.shibboleth.net/maven/releases/` 를 `org.opensaml`/`net.shibboleth` 그룹 한정으로 추가**(필수, fallback 아님 — 그룹 한정이라 그 외 의존성은 계속 Central, saml-sp 미사용 빌드 영향 0). poi/openpdf/sshd 의 "BOM 밖이라 버전 핀" 패턴과 **다름**(SAML 은 버전은 SS 가 관리하고 저장소만 추가). 해소 확인 `./gradlew :framework:framework-saml-sp:dependencies`.
- **SAML 체인은 메인 체인 "뒤"에 추가(2026-06-04)**: framework-security 메인 `securityFilterChain` 은 `@ConditionalOnMissingBean(SecurityFilterChain.class)` → SAML 체인을 먼저 등록하면 메인이 백오프돼 사라진다. `@AutoConfiguration(after = SecurityAutoConfiguration.class)` + 별도 `securityMatcher("/saml2/**","/login/saml2/**")` + 높은 우선순위(`@Order(HIGHEST_PRECEDENCE+50)`)로 둬 메인이 먼저 등록된 뒤 SAML 체인이 추가되고 매처 경로만 가로챈다(매처 밖은 메인 catch-all). SS7 SAML2 DSL = `saml2Login`/`saml2Logout`/`saml2Metadata`(셋 다 유효 — 초안의 `saml2Metadata` VERIFY 표시 해소). 신원은 `Saml2AuthenticatedPrincipal#getRelyingPartyRegistrationId()`+`getName()`(NameID)+`getAttributes()`(`Map<String,List<Object>>`).
- **SAML redis AuthnRequest 저장소는 후속(2026-06-04, 미구현 fail-fast)**: SP-initiated 흐름은 AuthnRequest↔Response 상관을 기본 HTTP 세션에 둔다 → 멀티 파드에서 authorize/ACS 가 다른 파드면 깨짐. 현 단계 해법=게이트웨이/인그레스 **스티키 세션**(핸드셰이크 수초). redis 공유는 `Saml2AuthenticationRequestRepository`(SS7: `load/save/removeAuthenticationRequest`, key=`Saml2ParameterNames.RELAY_STATE`) 구현이 필요한데 **`AbstractSaml2AuthenticationRequest`(Redirect/Post 하위타입) 직렬화가 난점**(Java 직렬화는 파드 버전 간 취약 → 빌더 기반 수동 (역)직렬화 권장) → 작성 환경 컴파일 불가라 검증 후 도입. 그때까지 `request-repository: redis` 설정 시 오토컨피그가 **시작 시 명확히 실패**(조용한 no-op 방지).

## 8. 문서 갱신 규칙
- **문서 역할**: `HANDOFF_SUMMARY.md` = *세션 한 장*(휘발) — 직전에 한 것·바로 다음 할 것·이번에 새로 밟은 함정. 매 세션 통째로 새로 씀. **`HANDOFF.md`(이 문서)** = *누적 정본*(영속) — 구조·원칙·함정·현재상태의 전체 그림. 구조/원칙/함정이 바뀔 때만 갱신.
- **새 세션 시작 워크플로**: 새 대화에 `HANDOFF_SUMMARY.md` 를 먼저 제시(즉시 이어하기) → 더 깊은 맥락 필요하면 `HANDOFF.md` 참조. (둘은 보완 관계: SUMMARY 로 연결, HANDOFF 로 전체 파악)
- **세션을 넘길 때마다** `HANDOFF_SUMMARY.md` 의 `<!-- 갱신 -->` 구간을 새로 쓴다. 양식은 **`HANDOFF_SUMMARY_TEMPLATE.md`** 를 복사해 `〈…〉` 만 채운다(B 절을 갱신 구간에 붙여넣기). 템플릿 A 절(고정 베이스라인: 완료 모듈·다음 후보·고정 함정·레시피)은 구조가 바뀐 세션에만 갱신. 새 모듈 추가 등 구조가 바뀐 세션이면 `HANDOFF.md`(1·6·7절)도 함께 갱신.
- 라이브러리/플러그인 **버전을 바꾸면** `gradle/libs.versions.toml`(단일 소스) → `STACK.md` 표 갱신.
- 구조/원칙/함정이 바뀌면 **이 문서**를, 사용법/데모가 바뀌면 `README.md` 를 갱신. 모듈 카탈로그/토글은 `docs/FRAMEWORK_MODULES.md`.
