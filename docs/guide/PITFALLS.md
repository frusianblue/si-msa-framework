# 함정·교훈 대장 (Pitfalls & Lessons Ledger)

> **이 문서는 자산이다.** 일반적으로 알려진 문제와 **이 프로젝트를 개발하며 실제로 부딪힌** 문제를 한곳에 누적한다. 같은 에러로 두 번 헤매지 않기 위한 기억 장치다.
> 인증/JWT 깊은 사례는 [`JWT_STATELESS_PITFALLS.md`](JWT_STATELESS_PITFALLS.md), 개발 표준은 [`DEVELOPER_GUIDE.md`](DEVELOPER_GUIDE.md). 세션별 원문 기록(verbose)은 [`../_internal/HANDOFF.md`](../_internal/HANDOFF.md) §6.

## 추가 규칙 (append-only)
- 빌드/런타임 에러나 함정을 만나면 **즉시 여기 한 줄 추가**한다(특히 재발한 것).
- 형식: 분류 태그 **[일반]**(널리 알려진 문제) 또는 **[겪음]**(이 프로젝트에서 실제 발생) + `증상 → 원인 → 해결`.
- 카테고리 안에서 추가. 카테고리가 없으면 새로 만든다. 기존 항목은 지우지 말고 갱신만.

태그: ★ = 여러 번 재발(특히 주의).

## 1. 빌드 · 의존성 (Gradle / BOM)

- ★ **[겪음] `compileOnly` 는 test 클래스패스로 전이 안 됨** — 증상: main 은 컴파일되는데 테스트에서 `package … does not exist`. 원인: test 소스셋은 main 의 `compileOnly`(web/jdbc/redis 등)를 못 봄. 해결: 그 의존을 `testImplementation` 으로 **재선언**. (프로젝트 전반에서 가장 자주 재발.)
- **[겪음] 모듈마다 `testImplementation 'org.springframework.boot:spring-boot-starter-test'` 필수** — 루트 `subprojects` 가 깔아주는 건 실행 런처뿐. 누락 시 `package org.junit.jupiter.api does not exist`.
- **[겪음] JUnit Platform launcher 누락(Gradle 9)** — 증상: 테스트 **발견 단계**에서 `OutputDirectoryCreator not available … unaligned versions`. 원인: `spring-boot-starter-test` 가 `junit-platform-launcher` 비전이 + Gradle 9 자동주입 안 함. 해결: 루트 `subprojects { testRuntimeOnly 'org.junit.platform:junit-platform-launcher' }`.
- **[겪음] 새 모듈 추가 시 `settings.gradle` include 누락** — `project '<X>' not found in project ':framework'`. 모듈 폴더 + 완성 `settings.gradle` 을 한 zip 으로 배포.
- **[겪음] BOM 밖 라이브러리는 버전 핀** — POI(`poi`)·OpenPDF(`openpdf` 2.0.2)·MINA SSHD(`sshd` 2.16.0)·Tika(`tika`)·ZXing(`zxing`)는 Spring BOM 밖 → `gradle/libs.versions.toml` 에 고정 + `implementation`(타입 비노출). 신규 추가 시 `STACK.md` 갱신.
- **[겪음] OpenSAML 은 Maven Central 에 없음** — `spring-security-saml2-service-provider` 가 OpenSAML 을 전이로 끌어옴(버전도 SS 관리). 해결: 버전 **핀 금지**(SS 관리), 루트 repositories 에 Shibboleth(`build.shibboleth.net/maven/releases`)를 `org.opensaml`/`net.shibboleth` **그룹 한정** 추가.
- **[겪음] SAS(인가서버)가 commons-logging 제외 → 런타임 `NoClassDefFoundError: LogFactory`** — 컴파일은 통과, 기동만 실패(SF7 은 spring-jcl 폐지). 해결: `implementation 'commons-logging:commons-logging:1.3.5'` 명시.
- ★ **[겪음] `.dockerignore` 부재 → 호스트 `build/`·`.gradle/` 누수로 이미지에 stale jar 가 담김** — 증상: 소스엔 신규 클래스(예: `SmokeClientSeeder`)가 분명히 있고 `./gradlew test` 도 통과하는데, 컨테이너 이미지로 빌드하면 그 클래스가 jar 에 없다(`grep -ac <Class> /application/app.jar` = 0). 파드도 정상 기동(에러 없음)하지만 신규 기능이 동작 안 함(예: OAuth2 클라이언트 0건). 원인: `deploy/docker/Dockerfile.build` 의 `COPY . .` 가 `.dockerignore` 부재로 호스트의 `services/*/build/`·`.gradle/`(증분·구성/빌드 캐시 스냅샷)까지 컨텍스트에 포함 → 컨테이너 Gradle 이 복사돼 들어온 옛 산출물/스냅샷을 **up-to-date** 로 보고 재컴파일을 건너뜀 → Dockerfile 2단계가 `build/libs/*.jar`(옛 jar) 를 그대로 이미지에 cp. ⚠️ `docker build --no-cache` 는 Docker **레이어** 캐시만 무효화 — 복사돼 들어온 Gradle 상태는 못 막는다(그래서 --no-cache 로도 안 고쳐짐). 진단: ① `docker run --rm <img> sh -c 'grep -ac <Class> /application/app.jar'`(0=빌드가 옛 jar 담음, 1=이미지 정상→배포 전파 문제) ② `kubectl get pod … -o jsonpath='{..image}{..imageID}'`(파드가 쓰는 태그/digest). 해결: **리포 루트 `.dockerignore`** 로 `**/build/`·`.gradle/` 등 제외(이미지 빌드를 소스에서 클린 컴파일로 hermetic 하게) + 일회성으로 `./gradlew :services:<svc>:clean` 후 재빌드. 일반 원칙: **컨테이너 이미지 빌드는 호스트 빌드 상태에 의존하면 안 된다 — 항상 소스에서 클린 빌드.** [§9 멀티서비스 배포와 동근원 — \"이미지가 코드를 안 따라옴\"]

- **[겪음] Spring Cloud Task 는 메인 Oakwood BOM(2025.1.1)에 없음 → 별도 BOM** — 증상: `spring-cloud-task-core` 버전 미해소. 원인: SCT 는 Spring Cloud 릴리스트레인 BOM 밖. 해결: `dependencyManagement` 에 `mavenBom 'org.springframework.cloud:spring-cloud-task-dependencies:5.0.1'` 별도 import + 카탈로그 `springCloudTask="5.0.1"`. ⚠️ **`spring-cloud-starter-task` 는 stream 바인더까지 끌어옴** → 의도적으로 starter 대신 `spring-cloud-task-core`(+결합 시 `spring-cloud-task-batch`)만 의존.

## 2. Jackson 3 / 직렬화

- ★ **[겪음] `com.fasterxml.jackson.core/databind` import 금지** — 이 스택은 **Jackson 3 (`tools.jackson.*`)**. import 시 컴파일 에러. 예외: `com.fasterxml.jackson.annotation`(애너테이션만) 유지 OK. 필터/인프라 단순 JSON 은 빈 주입 대신 **수기 직렬화**가 견고.
- **[겪음] Redis 값 직렬화에 `GenericJackson2JsonRedisSerializer` 금지** — Jackson 2 라 규약 위반. JDK 직렬화(`RedisSerializer.java()`)+`StringRedisTemplate` 수기. JSON 필요하면 앱이 `RedisCacheConfiguration` 빈으로 Jackson 3 구성.
- **[겪음] `FactorGrantedAuthority` 직렬화는 SS core Jackson 3 모듈이 지원** — `org.springframework.security.jackson.CoreJacksonModule`(allowIfSubType+mixin). 커스텀 직렬화 불필요.

## 3. Spring Boot 4 / SF7 패키지 이동 (★ 추측 금지 — 정본 소스 확인)

- **[겪음] autoconfigure 패키지 분리** — 모듈별 `org.springframework.boot.<module>.autoconfigure.*` 로 이동(jdbc/batch/quartz/mail). `@AutoConfiguration(afterName=...)` FQCN 확인.
- **[겪음] `EnvironmentPostProcessor` 이동** — `org.springframework.boot.env.*` → **`org.springframework.boot.EnvironmentPostProcessor`**. **코드 import + `spring.factories` 키 둘 다** 변경(키만 틀려도 조용히 미등록).
- ★ **[겪음] Spring Batch 6 인프라 아이템 패키지 대이동** — `ItemReader/ItemProcessor/ItemWriter/Chunk`·`JdbcCursorItemReader(Builder)`·`JdbcBatchItemWriter(Builder)`·`RepeatStatus` 가 **`org.springframework.batch.infrastructure.item.*`** 로 이동(5.x 까지의 `org.springframework.batch.item.*` 아님). 반면 `Job`/`Step`/`JobRepository`/`JobBuilder`/`StepBuilder`/`Tasklet`/`JobParameters` 등 core 는 `org.springframework.batch.core.*` 그대로(단 `Step` 은 `core.step.Step`, `JobOperator` 는 `core.launch.JobOperator`). `JobLauncher` deprecated → `JobOperator`. 추측 금지 — raw 소스로 FQCN 확인.
- **[겪음] `JdbcBatchItemWriterBuilder.beanMapped()` 는 JavaBean 게터(`getX()`) 필요 → record 안 됨** — record 접근자는 `x()` 형태라 `BeanPropertySqlParameterSource` 가 못 잡음(파라미터 null). 해결: 라이터 입력 타입은 **게터를 가진 일반 클래스**로(리더 입력은 record 가능). 또는 `columnMapped()`. **`FlatFileItemWriterBuilder.delimited().names(...)` 도 동일**(BeanWrapperFieldExtractor=게터 기반) — 추출 대상 타입은 게터 클래스.
- **[겪음] Batch6 파일 아이템 패키지 + FlatFile API 위치** — `FlatFileItemReader(Builder)`/`FlatFileItemWriter(Builder)`/`FlatFileHeaderCallback` = `org.springframework.batch.infrastructure.item.file(.builder)`; `FieldSet` = `...item.file.transform`; `FieldSetMapper` = `...item.file.mapping`. 리더는 `.delimited().names(...)` 후 `.targetType(Class)`(record/JavaBean 매핑) 또는 `.fieldSetMapper(fs -> ...)`(명시 파싱, LocalDate/BigDecimal 직접 변환 권장). 헤더 skip 은 `.linesToSkip(n)`.
- **[겪음] `FlatFileItemWriter.resource(...)` 는 WritableResource 요구 — `file:` 로케이션 문자열 주의** — `@Value("${..}") Resource` 로 `file:` 를 받으면 읽기 전용 `UrlResource` 로 해소될 수 있다. 경로 문자열을 받아 `new FileSystemResource(path)`(WritableResource 구현)로 직접 만들 것. 재실행 덮어쓰기는 `.shouldDeleteIfExists(true)`.
- **[겪음] `RunIdIncrementer` 패키지 이동(Batch6)** — `org.springframework.batch.core.job.parameters.RunIdIncrementer`(5.x `...launch.support` 아님). `new JobBuilder(name, repo).incrementer(new RunIdIncrementer()).start(step)` 로 동일 파라미터 재실행 시 새 JobInstance(`run.id` 증가) → "이미 완료" 거부 회피. `RepeatStatus` 는 `infrastructure.repeat.RepeatStatus`, `Tasklet`/`StepContribution` 은 `core.step(.tasklet)`, `ChunkContext` 는 `core.scope.context`.
- ★ **[겪음] Batch6 멀티스레드 청크 = 페이징 리더 필수(커서 불가)** — `.taskExecutor(...)` 로 청크를 병렬화하면 **커서 리더(`JdbcCursorItemReader`)는 안전하지 않다**(단일 ResultSet/커넥션 공유). `JdbcPagingItemReader`(페이지 단위)로 바꾸고 `sortKeys`(결정적 정렬, `Order`=`infrastructure.item.database.Order`) 지정 + 동시성에선 재시작 의미가 깨지므로 `saveState(false)`. `taskExecutor` 는 `AbstractTaskletStepBuilder` 에 있어 `SimpleStepBuilder` 가 상속.
- **[겪음] Batch6 로컬 파티셔닝 패키지/스코프** — `Partitioner` = **`org.springframework.batch.core.partition.Partitioner`**(`...partition.support` 아님), 반환 `Map<String, ExecutionContext>` 의 `ExecutionContext` = **`infrastructure.item.ExecutionContext`**(`putLong`). 매니저는 `new StepBuilder(name,repo).partitioner("workerStep", p).step(worker).gridSize(n).taskExecutor(...)`. 워커 리더는 `@StepScope`(=`core.configuration.annotation.StepScope`) + `@Value("#{stepExecutionContext['minId']}")` 로 파티션 경계 주입(파티션마다 새 인스턴스).
- **[겪음] Batch6 플로우 DSL 위치** — `Flow`=`core.job.flow.Flow`, `FlowBuilder`=`core.job.builder.FlowBuilder`. 조건분기 `.start(step).on("FAILED").to(x).from(step).on("*").to(flow).next(step).end().build()`; 병렬 `new FlowBuilder<Flow>(name).split(taskExecutor).add(flowA, flowB).build()`. `CompositeItemProcessor`=`infrastructure.item.support`, 빌더 `CompositeItemProcessorBuilder.delegates(p1, p2, ...)`(앞 출력=다음 입력).
- **[겪음] Batch6 저장프로시저 리더(PG refcursor) + record 친화 라이터** — `StoredProcedureItemReaderBuilder.procedureName(...).function().refCursorPosition(1).parameters(new SqlOutParameter("results", Types.REF_CURSOR)).rowMapper(...)`(빌더/리더 = `infrastructure.item.database(.builder)`, `SqlOutParameter`=`org.springframework.jdbc.core`). ⚠️ **PostgreSQL refcursor 는 트랜잭션 안에서만 유효** — 청크 스텝 트랜잭션 경계 안에서 호출/소비. 라이터에서 **record 를 그대로** 쓰려면 `beanMapped()`(게터 필요) 대신 `itemPreparedStatementSetter((item, ps) -> ...)`(위치형 `?`).
- **[겪음] actuator/micrometer 재편** — `MeterRegistryCustomizer` 가 `org.springframework.boot.micrometer.metrics.autoconfigure` 로.
- **[겪음] `HttpHeaders` 가 `MultiValueMap` 미구현(SF7)** — `containsKey`→`containsHeader`, `keySet`→`headerNames`, `forEach/entrySet` 제거.
- **[겪음] `ClientHttpRequestFactoryBuilder/Settings` 가 starter-web 컴파일 경로에 없음(Boot4 분리)** — RestClient 타임아웃은 spring-web `SimpleClientHttpRequestFactory` 로.
- **[겪음] Spring Session 프로퍼티 네임스페이스 변경(Boot4)** — `spring.session.redis.*` → **`spring.session.data.redis.*`**(autoconfigure 모듈 분리). 구 키는 조용히 무시됨(세션이 메모리로 떨어져 멀티 인스턴스에서 로그인 유실). `spring-session-data-redis` 는 `spring-boot-starter-data-redis` 동반 필요.
- **교훈**: 컴파일 미검증 환경이므로 FQCN/이동은 **공식 소스(GitHub raw 해당 버전 브랜치)** 로 확인하고 고친다. 추측 금지.

## 4. 오토컨피그 / 빈 등록

- ★ **[겪음] `ApplicationContextRunner` 가 `@Bean` 의 모든 파라미터/반환/중첩 타입을 로드** — `@ConditionalOnClass` 무관하게 introspect → 그 오토컨피그의 `compileOnly` 타입을 **전부** `testImplementation` 재선언해야 컨텍스트가 뜸(누락 시 `Failed to parse configuration class`). 운영은 ASM 으로 읽어 무관(테스트 전용 함정).
- **[겪음] `.imports` 등록 누락 = 죽은 코드** — 오토컨피그 클래스만 만들고 `META-INF/spring/...AutoConfiguration.imports` 에 안 넣으면 토글해도 안 켜짐(redis LoginAttempt 사례). 해결: 클래스 + `.imports` + 등록 가드 테스트(`.imports` 를 직접 읽어 단언).
- **[겪음] before-순서로 core 백오프 유도** — core 가 `matchIfMissing=true` 로 항상 등록하는 빈(CacheManager 등)을 대체하려면 `@AutoConfiguration(before=CacheAutoConfiguration)`. 동적 DataSource 는 `ImportBeanDefinitionRegistrar`(BDRPP 는 before-순서 못 지킴).
- **[겪음] 선택 백엔드는 중첩 `@Configuration static` + 클래스레벨 `@ConditionalOnClass`** — 톱레벨 `@Bean` 파라미터가 `compileOnly` 타입이면 Spring 이 클래스 전체를 먼저 로드해 메서드레벨 가드보다 먼저 깨짐.
- **[일반] 기능 모듈 토글 기본값 = OFF(opt-in)** — 대부분의 `framework.<x>.enabled` 는 `matchIfMissing` 없는 `havingValue="true"` 라 **명시 안 하면 비활성**(security 마스터만 `matchIfMissing=true`=ON 예외). README `끄는 법`/`켜는 법` 작성·문서화 시 이 기본값을 추측하지 말고 `@ConditionalOnProperty(matchIfMissing=?)` 로 확인. 3단 스토어 토글(`...store.type=memory|jdbc|redis`)은 redis 백엔드가 framework-redis 에 있을 때만 선택됨(`@ConditionalOnClass(StringRedisTemplate)` + `.imports`).
- ★ **[겪음] 두 번째 `SecurityFilterChain` 빈이 메인 체인을 조용히 억제** — 증상: webauthn/saml 같은 모듈을 켜면 main 보안 체인이 사라져 전 경로 무인증/오작동. 원인: framework-security 메인 체인이 `@ConditionalOnMissingBean(SecurityFilterChain.class)` 가드 → **아무 체인이나 먼저 등록되면** 메인이 백오프. 해결: 추가 체인 모듈은 ① `@AutoConfiguration(after = SecurityAutoConfiguration.class)` 로 **메인이 먼저 등록**되게 하고 ② 추가 체인에는 `@ConditionalOnMissingBean` 을 **달지 않으며** ③ `@Order`(고우선순위, 예 `Ordered.HIGHEST_PRECEDENCE + 50`) + `securityMatcher(경로 한정)` 로 path-한정 체인 + 메인 catch-all 공존. (saml-sp·webauthn 동일 패턴.)

## 5. 보안 / 인증 / JWT  → 깊은 사례는 [`JWT_STATELESS_PITFALLS.md`](JWT_STATELESS_PITFALLS.md)

- ★ **[겪음] 커스텀 `AuthenticationProvider` 가 `FactorGrantedAuthority` 누락 → OIDC `authenticationTime cannot be null`** — SS7 은 auth_time 을 인증 팩터 `issuedAt` 에서 산출. 해결: `FactorGrantedAuthority.fromAuthority(PASSWORD_AUTHORITY)` 부착 + roles 클레임에서 필터 제외.
- ★ **[겪음/일반] 멀티 인스턴스 + `type=memory` → 상태 미공유** — 토큰스토어·로그인잠금·멱등·락·MFA 챌린지 전부. 운영은 `redis`.
- **[겪음] stateless JWT 폐기 불가(로그아웃)** — jti 블랙리스트 + 짧은 TTL/회전. (5질문 ⑤)
- **[겪음] RBAC deny-by-default 아님** — `resources`/`role_resources` 매핑 없으면 인증 사용자 허용. 보호 자원은 매핑 명시 + `@PreAuthorize`.
- **[겪음/일반] `X-Forwarded-For` 위조 가능** — `login-id-and-ip` 잠금 키는 신뢰 프록시 환경만.
- **[겪음] JWT/AES 시크릿 prod 가드** — placeholder·약한키면 기동 실패(`JwtSecretSafetyGuard`/`AesMasterKeySafetyGuard`). 운영은 강한 키 env 주입.

MFA WebAuthn 2차는 동일 RP 빈을 재사용하므로 rpId/origin 이 자동 일관(별도 설정 없음).
- ★ **[겪음] AS 서명키 복호화 실패 ACTIVE 키가 부트스트랩을 막아 JWKS 0개 → token/jwks 500(겉으론 302/login)** — 증상: `/.well-known/openid-configuration` 은 200 인데 `POST /oauth2/token`(정상 `client_secret`)·`GET /oauth2/jwks` 가 **302 → `/login`**(로그인 폼으로 튕김). 틀린 시크릿은 401(client-auth 필터 자체 거부)이라 더 헷갈린다. 원인: `auth_signing_key` 에 ACTIVE 행은 있으나 그 개인키(`jwk_json`, `enc:` 마커 = AES 암호문)가 **현재 마스터키(`framework.crypto.aes-secret`/`AES_SECRET`)로 복호화 안 됨**(예: AES_SECRET 교체 후 옛 키만 잔존, DB(PVC)는 생존). `JdbcRotatingJwkSource.ensureBootstrapKey()` 가 `findNewestActive()!=null`(=행 존재)만 보고 **새 키 생성을 스킵**하는데, `loadFromDb()` 는 그 키를 복호화 실패로 건너뛰어 → 사용 가능 키 0개 → `IllegalStateException`→`JwtEncodingException`. 그 500 이 서블릿 ERROR 디스패치로 `/error` 포워딩되고 AS 가 아닌 일반 체인(`@Order(2)`)이 `/error` 를 `authenticated()` 로 막아 **302/login 으로 둔갑**(아래 항목). discovery 만 200 인 건 그것만 서명키를 안 건드리기 때문(셋의 차이가 진단의 결정타). 진단: 로그 `WARN JdbcRotatingJwkSource: 서명키 파싱/복호화 실패 — 건너뜀. kid=…` + `IllegalStateException: 사용 가능한 서명키가 없습니다(부트스트랩/복호화 실패 가능)`; `psql -d authdb -c "SELECT kid,status,left(jwk_json,16),length(jwk_json) FROM auth_signing_key;"`(head 가 `enc:` 면 암호문). 해결: ① 즉시 — `kubectl exec statefulset/postgres -- psql -U postgres -d authdb -c "DELETE FROM auth_signing_key;"` 후 `rollout restart deploy/auth-server`(부트스트랩이 현재 마스터키로 새 ACTIVE 키 1개 생성, 샌티 환경은 손실 없음). ② 코드 영구화 — `ensureBootstrapKey()` 를 "행 존재" 가 아니라 **"복호화 가능한 ACTIVE 존재"**(`hasUsableActiveKey()`: `RSAKey.parse(cipher.reveal(...))` 성공 여부)로 판정해, 옛 키가 복호화 불가면 새 ACTIVE 를 부트스트랩(복호화 불가 키는 `loadFromDb` 가 격리하므로 비파괴 보존). **일반 원칙: "키가 있는가" 와 "키를 쓸 수 있는가" 는 다르다 — 마스터키 교체 가능성이 있는 한 부트스트랩/헬스는 복호화 가능 여부까지 확인해야 한다.** [§5 `/error` 마스킹 항목과 한 묶음]
- ★ **[겪음] AS 일반 체인이 `/error` 를 막아 내부 500 을 302/login 으로 둔갑(디버깅 함정)** — 증상: AS 엔드포인트 내부 예외(위 서명키 등)가 본래 5xx 여야 하는데 `302 → /login;jsessionid=…`(+`Set-Cookie JSESSIONID`)로 응답 → 진짜 원인이 시큐리티/인증 문제처럼 보여 디버깅이 토끼굴로 샌다(이 한 함정이 §S3' 막판 반나절을 잡아먹음). 원인: 서블릿 컨테이너는 처리 중 예외 시 내부적으로 **ERROR 디스패치로 `/error` 에 포워딩**하는데, `AuthorizationServerConfig` 의 `@Order(2) defaultSecurityFilterChain`(AS 엔드포인트 매처에 안 걸리는 그 외 요청)이 `/error` 를 `anyRequest().authenticated()` + `formLogin` 으로 잡아 미인증 → 엔트리포인트 → 302/login. (`/actuator/**` 만 permitAll 이고 `/error` 는 빠져 있었음.) 진단 신호: 보안 WEB TRACE(`LOGGING_LEVEL_ORG_SPRINGFRAMEWORK_SECURITY_WEB=TRACE`)에 원 요청(`/oauth2/token`)이 아니라 **`Authorizing GET /error`** 가 찍힘 = 내부 포워딩 신호. 해결: `defaultSecurityFilterChain` 의 permitAll 에 **`"/error"` 추가**(`requestMatchers("/actuator/**", "/error").permitAll()`) → 내부 500 이 정직하게 5xx 로 드러나 다음 디버깅이 정확해진다. **일반 원칙: 커스텀 시큐리티 체인은 `/error`(+`/actuator/**`)를 permitAll 해 프레임워크 에러 디스패치가 인증 리다이렉트로 가려지지 않게 한다(에러를 마스킹하지 말 것).** [§5 / 위 서명키 항목과 한 묶음]
- **[일반] curl 기본 `Accept: */*` 는 `MediaTypeRequestMatcher(TEXT_HTML)` 에 매칭된다** — AS 멀티체인 정본 샘플은 `exceptionHandling.defaultAuthenticationEntryPointFor(LoginUrl, new MediaTypeRequestMatcher(MediaType.TEXT_HTML))` 로 "브라우저만 /login, API 는 401" 을 의도하지만, `text/html.isCompatibleWith(*/*)` 가 true 라 **curl(`*/*`) 도 TEXT_HTML 매처에 잡혀** API 인증실패도 401 대신 302/login 으로 보일 수 있다(매핑 순서상 TEXT_HTML 이 먼저면). 즉 curl 로 401 검증이 안 보여도 설정이 틀린 게 아닐 수 있음 — `-H 'Accept: application/json'` 로 재현/구분하거나, 토큰류엔 AS configurer 가 심는 `HttpStatusEntryPoint(401)` 매핑이 살아있는지(블랭킷 `authenticationEntryPoint(...)` 미설정)로 판단. ## 6. MyBatis / DB

- **[겪음] `@MapperScan` 에 SPI 섞인 패키지면 `annotationClass = Mapper.class`** — 미지정 시 `ConflictingBeanDefinitionException`(commoncode/file). 앱 전용 매퍼 패키지는 plain OK.
- **[겪음] H2(MODE=PostgreSQL) SQL 이식성** — `TIMESTAMPTZ`/`ON CONFLICT`(PG 전용)는 H2 파싱 실패 → `TIMESTAMP`+평문 INSERT 로 양립.
- **[겪음] 감사 로그 DB 적재 3요건** — ① 모듈 의존 ② `store.type=jdbc` ③ `audit_log` 테이블. 하나라도 빠지면 INSERT 실패를 삼키고 WARN 만.
- **[겪음] 멱등/락 JDBC = PK 유니크 + INSERT 충돌(`DataIntegrityViolationException`) 선점** — 트랜잭션/락 불요. UPSERT 금지(UPDATE→0행이면 INSERT)로 H2/PG/Oracle 양립.
- **[겪음] framework-mybatis 재사용 서비스는 `mybatis.mapper-locations: classpath*:mapper/**/*.xml` 필수** — 없으면 `Invalid bound statement (not found)`.
- ★ **[겪음] Spring Batch 6 에서 MyBatis 배치 = `mybatis-spring 4.0.0+` 필수** — `MyBatisPagingItemReader`/`MyBatisBatchItemWriter`(`org.mybatis.spring.batch`)가 **4.0.0 부터** 신패키지 `org.springframework.batch.infrastructure.item.*` 를 import(3.0.x 는 구 `org.springframework.batch.item.*` = Batch 5 전용 → Batch 6 과 비호환, 런타임 `NoClassDefFoundError`/컴파일 불가). 함정: `mybatis-spring-boot-starter` 는 현재 `mybatis-spring 3.0.5` 를 전이 → **`org.mybatis:mybatis-spring:4.0.0` 직접 선언**(highest-version-wins 로 3.0.5 대체). mybatis-spring 4.0.0 타깃 = mybatis 3.5.19 / Spring 7.0.1 / Spring Batch 6.0.0(우리 스택 정합). 페이징 select 는 `LIMIT #{_pagesize} OFFSET #{_skiprows}`(MyBatis 가 `_page/_pagesize/_skiprows` 주입), 리더/라이터는 statement id 문자열(`NS.selectTxnPage`/`NS.insertOut`)로 가리킴. 부트 스타터 없이 쓰면 `SqlSessionFactoryBean`(`org.mybatis.spring`) + `@MapperScan`(`org.mybatis.spring.annotation`) 직접 배선.

- **[일반/겪음] standalone kind 재부팅 — 무엇이 살아남나(dev overlay 기준)** — Docker Desktop 재시작 시 kind 노드(=Docker 컨테이너)가 같이 뜨고 etcd 상태(control-plane 컨테이너 fs)가 보존되므로 **모든 API 오브젝트(Deploy/StatefulSet/ConfigMap/Secret) 보존 + 컨트롤러가 파드 자동 재생성**(매니페스트 재적용 불필요). **DB 데이터 보존** = dev overlay postgres 는 emptyDir 가 아니라 **StatefulSet + PVC(`components/postgres-persistent`, `data-postgres-0` 5Gi, 기본 SC=local-path)** → PVC 가 노드 fs 에 남아 파드/노드 재시작 견딤(authdb/sidb/admindb + 시드 유지). ⚠️ **local overlay 는 emptyDir(휘발)** 라 정반대 — 재부팅/재생성 시 DB 초기화. **Grafana 대시보드도 ConfigMap → sidecar 재적재**. **손으로 다시 할 것 = `kubectl port-forward`**(호스트 프로세스, 클러스터 밖 → 리부트 시 사망, 재실행 필요). 단 kind 는 호스트 리부트 생존 **공식 미보장** — 가끔 control-plane wedge(노드 IP 변동 등) → 결정적 회복 = `00-cleanup.sh --teardown-sanity` 후 `01→…→06` 재구축. ⚠️ **`--teardown-sanity` 는 PVC 까지 삭제** → DB 데이터 남기려면 teardown 없이 Docker Desktop 만 재시작. 재기동 직후 앱이 postgres 준비 전 떠서 CrashLoop→회복은 정상(startup 프로브 ~150s). [삼단 토글 = §4 / DEVELOPER_GUIDE]\n- ★ **[겪음] auth-server prod 에 `Authenticator` 빈 없어 기동 실패 → 프로파일별 인증기 분리**: 증상: prod 부팅 시 `required a bean of type …security.auth.Authenticator`(또는 LoginService 연쇄). 원인: `LocalDemo` 의 demo 인증기가 `@Profile("local")` 이라 prod/dev/K8s 엔 `Authenticator` 빈이 없다(프레임워크는 인증 백엔드를 의도적으로 비워둠 — 프로젝트마다 다름). 해결: `user/ProdAuthenticatorConfig`(`@Profile("!local")` + `@Bean @ConditionalOnMissingBean(Authenticator.class)`)가 authdb `app_user` 기반 `DbAuthenticator` 등록. 함정 2개 동반: ① **인증기는 `@Component` 아니라 `@Configuration`+`@Bean`** 으로 — 컴포넌트 스캔 빈은 `@ConditionalOnMissingBean` 평가 순서가 불안정(@Bean 메서드에서만 신뢰). ② **seed 비번은 `{bcrypt}$2a$…` 포맷 필수** — `BcryptEnforcingPasswordEncoder`(framework-security 위임 인코더)가 `{noop}`/평문은 매칭 거부라, 평문이나 접두 없는 BCrypt 를 넣으면 항상 로그인 실패. - ★ **[일반] (kind 첫 배포 전 정적 점검에서 선포착) local overlay 이미지명 정합 — `images:` 는 tag만 바꾸면 name 이 안 바뀐다** — 증상(예상): `kubectl apply -k overlays/local` 후 4파드 전부 `ImagePullBackOff`(노드엔 `si-msa/<svc>:local` 만 적재돼 있는데 Deployment 는 `registry.example.com/si-msa/<svc>:local` 를 찾음 → IfNotPresent 가 노드에서 못 찾고 가짜 레지스트리로 pull 시도). 원인: kustomize `images` 변환기에서 `{name, newTag}` 만 주면 **tag 만 치환되고 name(=`registry.example.com/...`)은 그대로 남는다**. 반면 compose 가 실제 빌드/적재하는 이미지는 short name `si-msa/<svc>:local`(검증된 그린 산출물)이라 **렌더 이미지명 ≠ 노드 적재명**. base deployment 가 `registry.example.com/...:latest` 로 박혀 있어 dev/prod(실 레지스트리 push)는 정상이지만, **레지스트리 없는 로컬(kind/minikube)에서만** 터지는 정합 결함. 해결: local overlay 의 `images:` 에 **`newName: si-msa/<svc>` 를 함께** 지정해 가짜 접두어를 떼고 short name 으로 통일(`{name: registry.example.com/si-msa/gateway, newName: si-msa/gateway, newTag: local}` …×4). 그러면 compose 빌드 이미지를 **재빌드 없이 `kind load docker-image si-msa/<svc>:local` 만** 해서 그대로 쓴다. 점검: `kubectl -n si-msa get deploy -o jsonpath='{..image}'` 가 전부 `si-msa/...:local` 인지, `docker exec si-msa-control-plane crictl images | grep si-msa` 적재명과 일치하는지. **일반 원칙: 자기완결 로컬 클러스터엔 레지스트리가 없으므로 overlay 가 이미지 name 까지 노드 적재명과 1:1 로 맞춰야 한다(tag 만으론 부족).** [§8 운영 / 배포 정합]
- ★ **[겪음] Docker Desktop 의 kind 모드는 NetworkPolicy 를 집행한다 → 앱→postgres 차단으로 Flyway `Connect timed out`** — 증상: 이미지/스케줄 정상(파드 `desktop-worker` 배정 성공)인데 auth/user/admin 만 `CrashLoopBackOff`, 로그 `Caused by: java.net.SocketTimeoutException: Connect timed out` → `FlywaySqlUnableToConnectToDbException: Unable to obtain connection`(SQLState 08001). gateway 만 살아남음(DB 미사용). 원인: `base/common/networkpolicy.yaml` 의 `default-deny-ingress`(ns 전체 수신 차단)는 있는데 **postgres 수신 허용 규칙이 없다**(redis 는 `allow-redis-from-apps` 로 base 에 있지만, postgres 는 `overlays/local` 전용 파드라 base 엔 없음). **핵심 오해**: "kind = kindnet = NetworkPolicy 비집행" 은 **standalone kind 한정**이고, **Docker Desktop 의 내장 kind 모드는 정책을 집행하는 CNI** 라 로컬에서도 default-deny 가 실제로 막는다(증상이 `password authentication failed`/`UnknownHostException` 이 아니라 `Connect timed out`/08001 이면 인증·DNS 가 아니라 **L3/4 차단** = NetworkPolicy 신호). 해결: `overlays/local/postgres.yaml` 에 `allow-postgres-from-apps`(podSelector name=postgres, ingress from component=service, TCP 5432) 추가 → `default-deny` 유지하며 앱→PG 만 연다(redis 규칙과 동일 패턴). 즉시 확인/해소: `kubectl -n si-msa get networkpolicy`(postgres 허용 없으면 확정) → 위 규칙 `kubectl apply` 후 `kubectl -n si-msa rollout restart deploy/auth-server deploy/user-service deploy/admin-service`. **일반 원칙: NetworkPolicy 를 집행하는 클러스터에선 인-클러스터 의존(DB/캐시/메시지브로커) 각각에 default-deny 를 뚫는 allow 규칙이 1:1 로 있어야 한다.** [§8 운영 / 배포 정합]
- ★ **[겪음] postgres initdb 스크립트는 PGDATA 가 빈 최초 1회만 실행 → 기존 PG 에 DB/스키마 추가 안 됨** — 증상: admin-service `CrashLoop`, 로그 `Caused by: org.postgresql.util.PSQLException: FATAL: database "admindb" does not exist`(SQLState 3D000). 원인: `overlays/local/postgres.yaml` 의 `postgres-initdb`(authdb/sidb/**admindb** CREATE)는 `/docker-entrypoint-initdb.d` 규약상 **데이터 디렉터리가 비어있는 최초 부팅에만** 실행된다. admindb 줄을 추가하기 *전에* 이미 초기화돼 떠 있던 PG 파드(휘발 emptyDir 라도 파드가 살아있으면 PGDATA 유지)엔 authdb/sidb 만 있고 admindb 가 없다. (`relation already exists` 가 아니라 `database does not exist`/3D000 이면 이 신호 — 인증·연결이 아니라 **DB 자체 부재**.) 해결: ① 즉시 — `kubectl -n si-msa exec deploy/postgres -- psql -U postgres -c "CREATE DATABASE admindb OWNER siuser;"`(+`GRANT ALL ON SCHEMA public`) 후 `rollout restart deploy/admin-service`. ② 정공법 — `rollout restart deploy/postgres`(휘발 PG 재초기화 → initdb 가 admindb 까지 처음부터 생성, authdb/sidb 데이터도 초기화·로컬 무방). **신규 클러스터(`delete -k`→`apply -k`)는 initdb 가 admindb 까지 자동 생성**(검증됨) — 이 함정은 "스키마/DB 를 나중에 추가했는데 PG 가 이미 떠 있던" 타이밍 한정. **일반 원칙: initdb/엔트리포인트 시드는 1회성 — 런타임에 추가된 DDL 은 별도 마이그레이션(Flyway) 또는 수동 적용이 정석.** [§9 멀티서비스 배포]
- ★ **[일반/겪음] Docker Desktop 의 "Kubernetes(kind 모드)" 는 standalone kind 와 동작이 다르다** — 환경 식별: 컨텍스트 `docker-desktop`, 노드명 `desktop-control-plane/worker/worker2`, Cluster type=`kind`, `local-path-storage` ns 존재. 함정 묶음: ① **노드가 `docker ps` 에 안 보인다**(노드명은 k8s 노드 이름이지 도커 컨테이너 이름이 아님; WSL 에선 더더욱) → `docker exec desktop-worker …`/`ctr import` 식 적재 **불가**. ② **`kind` CLI/`kind load` 를 쓰지 않는다**(Docker Desktop 이 클러스터를 관리 — 별도 kind CLI 가 가리키지 못함; `kind create cluster` 동시 사용은 충돌). ③ **그러나 적재가 필요 없다** — `desktop-containerd-registry-mirror` 가 **호스트 docker 이미지를 노드 containerd 에 자동 노출**(standalone kind 의 "노드별 격리 containerd → kind load 필수" 와 정반대). 확인 1초: `kubectl run imgtest --image=si-msa/gateway:local --image-pull-policy=IfNotPresent --restart=Never --command -- sleep 60` → `Running` 이면 미러 동작(적재 불필요), `ImagePullBackOff` 면 폴백(로컬 레지스트리). **교훈: "kind=kindnet=비집행·kind load 필수" 통념은 standalone kind 한정이고, Docker Desktop kind 모드는 NetworkPolicy 집행 + 이미지 자동노출로 둘 다 반대다.** [§9 멀티서비스 배포 / 환경]
- ★ **[겪음] Docker Desktop kind 의 "이미지 자동노출" 편의의 이면 = 외부 레지스트리 직접 pull *불가*(구조적 제약)** — 증상: dev overlay apply 후 새 파드 `ImagePullBackOff`, `describe` Events 가 `failed to resolve reference "harbor.local/si-msa/<svc>:<tag>": ... GitOps 는 노드 pull 이 정상인 환경을 전제로 얹는 층. **일반 원칙: "노드가 레지스트리에서 pull" 은 쿠버네티스 불변 구조 — 배포도구(ArgoCD/Helm/kubectl)와 무관하며, 노드↔레지스트리 도달성은 클러스터/인프라 레이어에서 보장해야 한다.** [§9 / 환경 — 위 자동노출 항목과 한 묶음]

- ★ **[겪음] 같은 `:local` 태그 재빌드 → 노드 containerd 가 옛 digest 캐시한 채 `IfNotPresent` 로 새 이미지 무시** — 증상: `.dockerignore` 까지 고쳐 호스트 이미지엔 신규 클래스가 있는데(`docker run --rm --entrypoint sh <img> -c 'grep -ac <Class> /application/app.jar'` ≥1), `kubectl delete pod` 로 재생성해도 **파드 jar 는 여전히 0**·기능 안 됨. 원인: 로컬 overlay 가 고정 태그 `:local` + (기본)`imagePullPolicy: IfNotPresent` 라, 노드 containerd 에 그 **태그**가 이미 있으면(옛 digest) kubelet 이 pull 을 건너뛰고 캐시본을 재사용한다(같은 태그·다른 내용 두 이미지 공존). `delete pod` 는 새 컨테이너를 만들 뿐 이미지 재해석을 강제하지 않는다. 진단(결정적): `kubectl -n si-msa get pod -l app.kubernetes.io/name=<svc> -o jsonpath='{..imageID}'`(파드 사용 digest) vs `docker images --no-trunc -q si-msa/<svc>:local`(호스트 새 digest) — **다르면 노드 캐시 확정**. 해결: **유니크 태그로 우회** — `docker tag si-msa/<svc>:local si-msa/<svc>:local-N` → `overlays/local/kustomization.yaml` 의 해당 `newTag` 를 `local-N` 으로 → `kubectl apply -k …`. ⚠️ 흔한 오해: `docker run <img> sh -c '...'` 는 ENTRYPOINT(`java -jar`)가 인자를 먹어 **앱이 그냥 떠버린다**(grep 안 됨) → 반드시 `--entrypoint sh <img> -c '...'`. - ★ **[겪음] 가변 `:dev` 핀 + `kubectl set image :<sha>` 덮어쓰기 = git↔클러스터 드리프트(선언/명령 혼용)** — 증상(잠복형): dev overlay 는 `:dev` 로 박혀 있는데 CI deploy 가 `apply -k` 직후 `kubectl set image deployment/<svc> app=...:<git-sha>` 로 덮어, **git 의 진실(`:dev`)과 실제 뜨는 이미지(`:<sha>`)가 영구 불일치**. 부작용 3종: ① overlay 의 `:dev` 는 한 번도 무엇이 뜨는지를 지배하지 못하는 **죽은 태그**(매 빌드 push 낭비), ② 누가 `set image` 한 줄을 빼거나 수동으로 `apply -k` 만 하면 `:dev`+`IfNotPresent` 조합이 노드 캐시의 **옛 `:dev` 를 재pull 없이 조용히 띄움**(stale 사고 — 과거 `:7e935d6` 손핀 어긋남이 같은 뿌리), ③ "적용한 것 ≠ 뜨는 것"이라 GitOps/그린-박제 원칙 위반. 원인: **불변 sha 를 명령형으로 사후 주입**하면서 선언형 overlay 가 가변 태그를 들고 경쟁. 해결(B 결정 2026-06-07): **무엇이 뜨는지의 단일 진실 = 불변 git-sha 태그**, 그 태그를 **apply 시점에 overlay 에 declarative 로 주입**한다. (1) overlay `images.newTag` 를 sentinel `__GITSHA__` 로 둠 → **미주입 apply 는 ImagePullBackOff 로 fail-loud**(②의 조용한 stale 를 구조적으로 차단). (2) 주입은 공용 헬퍼 `deploy/k8s/pin-image-tag.sh <overlay-dir> <tag>` 한 곳에서만(CI·수동 트랙 공용, sed 기반·kustomize 바이너리 불요·멱등). (3) **체크아웃 워크스페이스만 치환(되커밋 X)** — CI 는 잡 워크스페이스, 수동(`03-dev-overlay-up.sh`)은 in-place 치환→apply→작업트리 복원(overlay 는 `../../base` 상대참조라 임시복사 apply 불가). (4) Kaniko/compose push 는 **`:<sha>` 단일 태그**(가변 `:dev` destination 제거). (5) `07-reboot-recover.sh` 재적재도 하드코딩 `:dev` 대신 **살아있는 Deployment 의 image ref 를 읽어 그 태그로** kind load. ⚠️ 단일 서비스 빠른 반복은 `redeploy.sh`(소스 **콘텐츠 다이제스트** 태그)가 이미 같은 원칙(불변 태그)을 구현 — 미커밋 변경까지 추적해야 할 땐 그쪽. ✅ **prod overlay 도 전환 완료(2026-06-08)** — `:latest` ×4 → `__GITSHA__`, `imagePullPolicy: Always` 패치 제거(전제였던 가변 `:latest` 소멸 → base `IfNotPresent` 상속). 단 prod 는 dev 와 **핀 주입 방식이 반대**(아래 GitOps 항목). **일반 원칙: 컨테이너 이미지의 "무엇이 뜨는가"는 불변 태그로 declarative 하게 한 곳에서 핀한다 — 가변 채널 태그를 배포 진실로 쓰거나, 선언형 매니페스트를 명령형 `set image` 로 사후에 덮지 않는다.** [§9 / CI·CD / 배포 정합]
- ★ **[일반] push-CD(워크스페이스 핀, 되커밋 X) ↔ pull-GitOps(커밋 핀) — 핀 주입 방식이 반대다** — dev(현행)=push-CD: CI 가 *체크아웃 워크스페이스에서만* `__GITSHA__` 를 치환(되커밋 X)하고 자기가 `kubectl apply`. git 엔 sentinel 이 영존한다. 그런데 **ArgoCD(prod)=pull-GitOps 는 단일 진실이 "git 에 커밋된 상태" 그 자체**라, 워크스페이스 치환을 모른다 → dev 의 "되커밋 X" 규칙을 그대로 prod 에 복사하면 ArgoCD 가 git 의 `:__GITSHA__` 를 보고 **영구 ImagePullBackOff** 로 reconcile 한다. 해결: **prod 는 핀을 git 에 커밋**한다(`kustomize edit set image .../<svc>=...:<sha>` 또는 pin-image-tag.sh 후 git commit/push) → ArgoCD 가 그 diff 를 sync. git 기본값을 `__GITSHA__` sentinel 로 두는 fail-loud 는 dev·prod 공통(promote 전 prod 는 sync/pull 실패로 stale 차단). ⚠️ **ArgoCD 는 노드→레지스트리 pull 을 풀어주지 않는다**(위 "노드 pull = 불변 구조" 항목과 한 묶음) — CD 도구는 매니페스트를 *apply* 할 뿐, pull 은 언제나 kubelet+노드 containerd. **일반 원칙: 배포 모델이 push(명령형 apply)냐 pull(git=진실)이냐에 따라 "불변 태그를 git 에 커밋하느냐 워크스페이스에서만 치환하느냐"가 갈린다 — pull-GitOps 는 무조건 커밋이 진실이다.** [§9 / CI·CD / 배포 정합 — 2026-06-08 ArgoCD prod 설계]

(2) **운영 애드온(ingress-nginx·metrics-server·cert-manager·CSI) = 워크로드가 도는 그 클러스터마다** — "부가는 컨트롤플레인에" 통념의 함정: ingress 는 트래픽 받는 클러스터에, metrics-server 는 HPA 도는 클러스터에 있어야 함(hub 에 몰면 안 됨). (3) **관측 = 워크로드 클러스터마다 Prometheus agent + 중앙 Grafana 는 hub**(분산형, finance/멀티클러스터 표준). (4) **DB·Redis = K8s 밖 독립** — 현장(특히 finance/공공 SI)은 DB 를 앱과 같은 클러스터에 두는 경우가 거의 없다. 대개 전용 서버/RDS/별도 보안존(망분리)에서 DBA 가 백업·HA·감사를 앱 라이프사이클과 분리 관리. 서비스는 그냥 `DB_URL` 로 그 엔드포인트에 접속할 뿐이라 cross-cluster 메시 같은 부담을 부르지 **않는다**(클러스터 경계는 평범한 네트워크 도달성 문제). prod overlay 가 `prod-postgres.internal` 로 외부를 가리키는 게 이미 이 모델 — dev/local 의 인-클러스터 `components/postgres-persistent` 는 *검증 편의*였지 운영 모델이 아니었다. **kind 리허설**: 데이터를 굳이 kind 클러스터로 만들지 말고 **우분투 도커 postgres/redis 컨테이너를 `--network kind` 로 붙여**(`DB_URL=jdbc:postgresql://<컨테이너명>:5432/…`) "DB=K8s 밖"을 더 정직히·가볍게 재현(클러스터 추가의 부대비용=별도 kubeconfig/CNI/DNS+cross-cluster 도달+ArgoCD 추가등록 회피). kind 멀티클러스터는 클러스터마다 별도 docker 네트워크라, hub→워크로드 API 도달은 kubeconfig 의 server 주소를 컨테이너 IP 로 등록해야 한다. **일반 원칙: 클러스터는 폭발반경·라이프사이클 경계로 가르고(CI/CD·데이터·워크로드), 애드온은 "그 기능이 필요한 클러스터"에 둔다 — DB 는 K8s 밖이 기본값.** [§9 / 운영 토폴로지 — 2026-06-08 prod 설계]
- ★ **[겪음] Kaniko 는 컨테이너당 executor 1회 모델 — 한 컨테이너 루프 다중빌드 시 2회차 `sh` 실패** — 증상: Jenkins kubernetes 에이전트의 단일 `kaniko` 컨테이너에서 4서비스를 `.each { sh "/kaniko/executor …" }`(또는 `for svc`) 로 돌리면 **1번째(gateway)는 성공**, 2번째 `sh` 스텝이 `Process exited immediately after creation` 으로 실패(gateway 만 빌드, 3서비스 누락). 원인: **kaniko executor 는 대상 이미지를 빌드하며 자기 컨테이너의 루트FS(`/`)를 그 이미지로 덮어쓴다** → `/busybox/sh`(:debug 이미지 셸) 가 클로버됨 → 다음 `container('kaniko')` `sh` 스텝이 셸을 못 띄움. 해결: **서비스별 kaniko 컨테이너 분리** — podTemplate 에 `kaniko-gateway/-auth-server/-user-service/-admin-service` 4개를 두고 각 `container("kaniko-${svc}")` 블록에서 executor **1회만** 실행(컨테이너가 다르면 셸도 매번 새것 → 클로버-재사용 자체가 없음). ★ **순차(병렬 금지)**: `Dockerfile.build` builder 스테이지는 SERVICE 무관(4 bootJar 한 번에) → 순차면 첫 빌드만 풀 Gradle, 나머지 3개는 `--cache-repo`(레지스트리)에서 builder 레이어 재사용(런타임 스테이지만). **병렬이면 4개가 동시에 cache-miss → Gradle 풀빌드 4회**(느림 + Docker Desktop 메모리 급증) — 공유 builder 캐시 설계를 무력화. 대안(미채택): 단일 kaniko + `--cleanup` + 단일 sh 루프(컨테이너 1개로 가볍지만 `--cleanup` 안정성 편차). ⚠️ 서비스 추가 시 **podTemplate 컨테이너 + 빌드 루프 리스트 동시 갱신**(둘 중 하나만 빠지면 `container not found` 또는 빌드 누락). [§9 / CI·CD]
- ★ **[겪음] CI deploy 단계 2종 결함 — (A) jenkins-deployer 가 클러스터 스코프 `namespaces` patch 불가, (B) ServiceMonitor CRD 가 코어 apply 를 막음** — 증상: 빌드/push 는 성공(4서비스 :<sha> Harbor 적재)했는데 deploy 단계 `kubectl apply -k overlays/dev` 가 일부 객체(ingress·networkpolicy)만 created 후, ① `no matches for kind "ServiceMonitor" in version monitoring.coreos.com/v1 — ensure CRDs are installed first`, ② `Error from server (Forbidden): namespaces "si-msa" is forbidden: User "system:serviceaccount:jenkins:jenkins-deployer" cannot patch resource "namespaces" in API group "" in the namespace "si-msa"`. **"네임스페이스 없음"이 아니다 — ns 는 09 가 선생성(존재함). apply 가 base 의 Namespace/si-msa 객체에 3-way merge patch 를 거는데 권한이 없는 것.** 원인: (A) jenkins-rbac 의 `edit` 는 **RoleBinding(네임스페이스 한정)** 이라 클러스터 스코프 리소스(Namespace)엔 미적용. (B) `base/common/servicemonitor.yaml` 의 SM 은 Prometheus Operator(kube-prometheus-stack) 설치돼야 존재하는 CRD 인데, dev overlay 가 그걸 그대로 적용(이전엔 "05 먼저 실행" 전제로 $patch:delete 를 뺐음) → operator 미설치 클러스터(CI/standalone 기본)에서 apply 전체 실패. 해결: **(A) ClusterRole+ClusterRoleBinding 로 `namespaces` get/patch 부여하되 `resourceNames: [si-msa]` 로 그 한 객체만**(최소권한; 생성은 09 admin 이 선보장 → CI 는 patch 만). jenkins-rbac.yaml 에 추가 후 **admin 컨텍스트로 재apply**(클러스터 스코프 RBAC 생성은 cluster-admin 필요 — jenkins-deployer 자신은 못 만든다). **(B) dev overlay 도 local 과 동일하게 SM 을 `$patch:delete`** → 코어 apply(CI·수동·standalone)는 operator 비의존. **관측의 SM 단일 소유자 = 05-prometheus-stack.sh**(operator 설치 후 `kubectl -n si-msa apply -f base/common/servicemonitor.yaml` 직접 — dev overlay 재적용 의존 제거). **일반 원칙: ① CI 배포 SA 는 자기 네임스페이스 객체(클러스터 스코프)에 한해 명시적 ClusterRole 이 필요하다(namespaced edit 으론 부족). ② 옵셔널 operator CRD 에 의존하는 리소스(ServiceMonitor 등)는 코어 앱 배포에 넣지 말고 해당 add-on 스크립트가 단독 소유한다 — 관측은 배포의 선결조건이 아니어야 한다.** [§9 / CI·CD / 배포 정합]
- ★ **[겪음/일반] Grafana·Prometheus 를 B안 ingress 로 노출 시 함정 3종** — standalone kind 의 ingress-nginx(control-plane hostPort 80/443) 뒤에 kube-prometheus-stack 의 Grafana/Prometheus 를 `grafana.local`/`prometheus.local` 로 붙일 때: ① **ssl-redirect 끄기 필수** — ingress-nginx 는 기본 HTTP→HTTPS 308 리다이렉트 → 인증서 없는 평문 접속이 깨진다. ingress annotation `nginx.ingress.kubernetes.io/ssl-redirect: "false"`(Harbor `ssl-redirect=false` 와 동일 근거). ② **Grafana 는 `server.domain`+`root_url` 을 호스트에 맞춰야** 로그인 리다이렉트/생성 링크가 정확(`grafana.ini: server: {domain: grafana.local, root_url: "%(protocol)s://%(domain)s/"}`). 서브패스(/grafana)면 `serve_from_sub_path: true` 도 필요하나 호스트 루트면 불요. Prometheus 는 `prometheusSpec.externalUrl: http://prometheus.local`. ③ **노드 이름해소 불요(Harbor 와의 결정적 차이)** — Grafana/Prometheus 는 노드가 **pull 하는 대상이 아니라** 호스트 브라우저가 보는 HTTP 서비스라, Harbor 처럼 노드 certs.d/노드 /etc/hosts/split-horizon 이 **전혀 필요 없다**. **호스트(Windows+WSL) hosts 한 줄**(`127.0.0.1 grafana.local prometheus.local`)이면 끝. 설치는 helm values(`monitoring-values.yaml`)로 grafana/prometheus.ingress + grafana.ini 를 주고 `05` 가 `-f` 로 머지(재실행=helm upgrade 멱등). 점검: `curl -H "Host: grafana.local" http://localhost/api/health`(hosts 무관, ingress 라우팅만 검증). **일반 원칙: ingress 노출 대상이 '노드가 pull 하는 이미지 레지스트리'냐 '호스트가 보는 웹 UI'냐로 이름해소 범위가 갈린다 — 후자는 호스트 hosts 한 줄로 족하다.** [§9 / 관측 / 배포 정합]

## 10. 배치 · 태스크 · 스케줄 (Batch · Task · Scheduling)

- **[일반] SCDF(Spring Cloud Data Flow) 서버 본체 = OSS EOL(2025-04)** — Broadcom 이 OSS 중단(GitHub `spring-attic` 아카이브). 마지막 OSS 2.11.x(Boot2/3), 이후 Tanzu 상용 전용. ⚠️ 새 프로젝트에 SCDF **서버**를 깔지 말 것. 반면 **Spring Cloud Task 라이브러리(5.0.1)는 생존 + Boot4 네이티브** — 실행이력 영속용으로 단독 사용 가능(SCDF 불필요).
- **[겪음] `@EnableTask`(=`@EnableFrameworkTask`) 없으면 실행이력이 시작도 안 됨** — `framework.task.enabled=true` 만으론 부족. SCT 는 부트 클래스에 `@EnableTask` 가 있어야 `TaskRepository`/`TaskExecutionListener` 인프라를 깬다(`SimpleTaskAutoConfiguration` 은 matchIfMissing 으로 TaskRepository 는 만들지만, 실행 생명주기 기록은 애너테이션이 트리거).
- **[겪음] SCT/Batch DDL 자동생성 끄고 Flyway 선반영** — 운영은 스키마를 마이그레이션으로 관리. `spring.cloud.task.initialize-enabled=false`(SCT 자동 DDL off, canonical) + `spring.batch.jdbc.initialize-schema=never`(Batch 자동 DDL off) + Flyway 로 `TASK_*`·`BATCH_*` 테이블 선반영(실소스 DDL: SCT 5.0.1 PostgreSQL = TASK_EXECUTION/_PARAMS/TASK_TASK_BATCH/TASK_SEQ/TASK_LOCK; Batch 6.0.3 = BATCH_JOB_INSTANCE/EXECUTION/_PARAMS/STEP_EXECUTION/(STEP|JOB)_EXECUTION_CONTEXT + 3 시퀀스).
- **[겪음] 작업 실패가 컨테이너 종료코드로 안 나옴** — k8s CronJob 은 종료코드로 성공/실패를 판정. `spring.cloud.task.batch.fail-on-job-failure=true` + 부트 `main` 에서 `System.exit(SpringApplication.exit(ctx))` 로 종료코드 전파 + Dockerfile `ENTRYPOINT exec`(쉘 래핑 금지)이라야 Job 실패→exit≠0→CronJob `backoffLimit` 재시도가 동작.
- **[겪음] 다중 인스턴스에서 같은 태스크 중복 실행** — `spring.cloud.task.single-instance-enabled=true`(기본 false) 로 TASK_LOCK 기반 단일 인스턴스 보장. CronJob 은 `concurrencyPolicy: Forbid`(겹침 방지) + `restartPolicy: Never`(Job 컨트롤러가 재시도 관리)도 함께.
- **[일반/겪음] Quartz 모니터링 UI 는 모델 B(framework-batch)·JDBC JobStore 전용** — UI 도구는 Quartz 스케줄러 테이블을 들여다본다. 기본 RAM JobStore 면 외부 도구가 볼 게 없다 → `spring.quartz.job-store-type=jdbc` + `org.quartz.jobStore.isClustered=true` + PostgreSQLDelegate + QRTZ_* 11테이블(DDL `deploy/db/quartz/`). run-once 태스크(모델 A)엔 Quartz UI 개념 자체가 없음.
- **[겪음] QuartzDesk 는 상용(OSS 아님)** — 무료 *Lite* 만 기능제한 제공, Standard/Enterprise 유료. 본 스택(Boot4/Java21) OSS 후보는 **`fabioformosa/quartz-manager`**(Apache, REST+임베드 UI, Boot 3.5/4.0 호환). Quartzmin/CrystalQuartz/quartznet-admin 은 **Quartz.NET 전용 → 부적합**. (가이드 `docs/guide/BATCH_SCHEDULING_AND_UI.md`)
